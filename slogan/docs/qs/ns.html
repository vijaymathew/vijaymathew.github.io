Modularity

A function can refer to variable bindings in the environment that created it. In the following program make_counter is a function that returns a function. The returned function maintains a reference to the argument of make_counter as its own "private state":

slogan> function make_counter(n) 
         fn(message) 
          match (message) 
           !inc -> { n = n + 1; 
                     n }, 
           !dec -> { n = n - 1; 
                     n }, 
           !get -> n; 

slogan> define c1 = make_counter(10); 
slogan> define c2 = make_counter(100); 
slogan> c1(!inc); 
=> 11
slogan> c1(!inc); 
=> 12
slogan> c2(!inc); 
=> 101
slogan> c2(!dec); 
=> 100
slogan> c1(!get); 
=> 12

There is a shorter dot-notation for "sending messages" to functions:

slogan> c2.inc; 
=> 101
slogan> c2.inc; 
=> 102
slogan> c2.get; 
=> 102

A function bound together with its defining environment is known as a closure. If you are familiar with Object Oriented Programming, you would like to call it an object. A function that creates a closure (in our example, make_counter) takes on the role of a class.

Namespaces allow to collect definitions into logical groups. They can also be used to avoid name conflicts as shown in the following program:

slogan> namespace a; 
slogan> function f() "hello from namespace - a"; 
slogan> define x = 100; 
slogan> namespace ; 

slogan> namespace b; 
slogan> function f() "hello from namespace - b"; 
slogan> define x = 200; 
slogan> namespace;

Names bound in a namespace are made visible by importing the namespace:

slogan> import a; 
slogan> f(); 
=> hello from namespace - a
slogan> x; 
=> 100

slogan> import b; 
slogan> f(); 
=> hello from namespace - b
slogan> x; 
=> 200

Import has options that allow us to import bindings from both namespaces at the same time without name conflicts. This is explained in the manual.

Large programs will spread across multiple source files. A Slogan source file can be imported using the load statement. It is customary to save Slogan scripts with the ".sn" extension. But it is better to avoid the extension while passing the script name to load. This allows Slogan to load the compiled object file, if that is available. A simple program that consists of two scripts is shown below:

// file: a.sn
function say_hello(name) string_append("hello ", name); 
// file: b.sn
load "a";	    
println(say_hello(second(command_line())));

The path passed to the load statement can contain place-holders that refer to environment variables. These variables will be dynamically expanded by the script loader. For instance, imagine you keep all your code libraries under /home/me/slogan/lib. If you point the environment variable SLOGAN_LIB to that directory, you can have load statements like the following in your code: 

load "$SLOGAN_LIB/mylib";

Now let us compile the source files to create a standalone executable. Before invoking the compiler from the REPL, make sure you have exported the environment variable SLOGAN_ROOT to point to the directory where you have extracted the Slogan source repository. For example, on GNU/Linux, if you have downloaded Slogan to /home/me/slogan, you can export the environment variable as:

$ export SLOGAN_ROOT=/home/me/slogan

Now you can restart the REPL and compile the program:

slogan> compile("b.sn", exe = true);

This will produce the executable binary named "b", which you can run directly from the shell. A sample invocation will look like:

$ ./b vijay
hello vijay
