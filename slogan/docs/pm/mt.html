Multitasking

Slogan supports the execution of multiple parallel tasks (or threads of execution). These tasks are managed entirely by Slogan's runtime and are not related to the host operating system's threads. The current implementation of Slogan does not take advantage of multiprocessors (i.e. at most one thread is running).

Memory coherency

Read and write operations on the store (such as reading and writing a variable, an element of a array or a string) are not atomic. It is an error for a task to write a location in the store while some other task reads or writes that same location. It is the responsibility of the application to avoid write/read and write/write races through appropriate uses of the synchronization primitives.

Concurrent reads and writes to ports are allowed. It is the responsibility of the implementation to serialize accesses to a given port using the appropriate synchronization primitives.

function: current_task() 
returns: the current task.
function: is_task(obj) 
returns: true if obj is a task object, false otherwise.
function: task(f, @key args, suspended, name = void, group) 
returns: a new task.

The function f is executed in the new task. If args are not provided or is false, f is treated as a no-argument function. Otherwise the list of objects args is passed to f when it is called. F should take as many arguments as there are elements in args. The Slogan interpreter treats calls to task specially allowing f to be an arbitrary expression. In such cases, a function is implicitly created with f as its body. If suspended is true the task is created in the suspended state. It has to be started with a call to task_run. If provided, name and group identifies the task's name and the group it belongs to.

function: root_task(f, @key args, suspended, name = void, 
                    group, input_port = current_input_port(), 
                    output_port = current_output_port()) 
returns: a new task.

This function is similar to task except the created task does not inherit the dynamic environment from the current task and the base priority is set to 0, the priority boost is set to 1.0e-6, and the quantum is set to 0.02. The dynamic environment of the task has the global bindings of the parameter objects, except current_input_port which is bound to input_port, current_output_port which is bound to output_port, and current_directory which is bound to the initial current working directory of the current process. 

function: task_run(t) 
returns: the task t.

Makes the task t runnable.

slogan> define a = task(fn (x, y) showln(x + y), args = [10, 20], suspended = true); 
slogan> task_run(a); 
-> 30
function: task_yield() 
returns: void

This function causes the current task to exit the running state as if its quantum had expired.

function: task_sleep(timeout) 
returns: void

This function causes the current task to wait until the timeout is reached. This blocks the task only if timeout represents a point in the future. It is an error for timeout to be false.

Timeout can be exact integers to indicate seconds or fractional numbers to indicate sub-second timeouts.

function: task_terminate(t) 
returns: void

This function causes an abnormal termination of the task t. If the task is not already terminated, all mutexes owned by it become unlocked/abandoned and a terminated-exception object is stored in the its end-exception field. If t is the current task, task_terminate does not return. The actual termination of the task will occur at some point between the calling of task_terminate and a finite time in the future (an explicit task synchronization is needed to detect termination, see task_join).

function: task_join(t, @optional timeout, timeout_val) 
returns: timeout_val.

This function causes the current task to wait until the task t terminates (normally or not) or until the timeout is reached if timeout is supplied. If the timeout is reached, this function returns timeout_val if it is supplied, otherwise a join-timeout-exception object is raised. If the task terminated normally, the content of the end-result field is returned, otherwise the content of the end-exception field is raised.

function: task_send(t, msg) 
returns: void

This function adds the message msg at the end of the mailbox of thread t.

function: task_receive(@opional timeout, default)
function: task_messages_next(@opional timeout, default) 
returns: a message from the current task's mailbox.

The task_receive function advances the mailbox cursor of the current task to the next message, removes that message from the mailbox, rewinds the mailbox cursor, and returns the message. When timeout is not specified, the current task will wait until a message is available in the mailbox. When timeout is specified and default is not specified, a mailbox-receive-timeout-exception object is raised if the timeout is reached before a message is available. When timeout is specified and default is specified, default is returned if the timeout is reached before a message is available.

Task_messages_next behaves like task_receive except that the message remains in the mailbox and the mailbox cursor is not rewound.

function: task_messages_rewind(@opional remove_last_read)
returns: void

This function rewind the mailbox cursor of the current task so that the next call to task_messages_next and task_receive will access the oldest message in the mailbox. Additionally if remove_last_read is true the message most recently accessed by a call to task_messages_next will be removed from the mailbox. When task_messages_next has not been called since the last call to task_receive or task_messages_rewind, a call to task_messages_rewind with remove_last_read set to true only resets the mailbox cursor (no message is removed).

Mutexes
function: mutex(@opional name = void)
returns: a new mutex in the unlocked/not-abandoned state.
function: is_mutex(obj)
returns: true if obj is a mutex, false otherwise.
function: mutex_name(m)
returns: the name of the mutex m.
function: mutex_set_data(m, obj)
returns: void

Sets the data field of the mutex to the arbitrary object obj.

function: mutex_data(m)
returns: the arbitrary object in the data field of the mutex m.
function: mutex_state(m)
returns:  information about the state of the mutex m.

Possible states are:

  • task T: the mutex is in the locked/owned state and task T is the owner of the mutex.
  • !not_owned: the mutex is in the locked/not-owned state.
  • !abandoned: the mutex is in the unlocked/abandoned state.
  • !not_abandoned: the mutex is in the unlocked/not-abandoned state.

function: mutex_lock(m, @optional timeout, task)
returns: either true or false. (see below for more details).

This function locks the mutex m. If the mutex is currently locked, the current task waits until the mutex is unlocked, or until the timeout is reached if timeout is supplied. If the timeout is reached, mutex_lock returns false. Otherwise, the state of the mutex is changed as follows:

  • if task is false the mutex becomes locked/not-owned,
  • otherwise, let T be task (or the current task if task is not supplied),
    • if T is terminated the mutex becomes unlocked/abandoned,
    • otherwise mutex becomes locked/owned with T as the owner.

After changing the state of the mutex, an abandoned-mutex-exception object is raised if the mutex was unlocked/abandoned before the state change, otherwise mutex_lock returns true. It is not an error if the mutex is owned by the current task (but the current task will have to wait).

function: mutex_unlock(m, @optional monitor, timeout)
returns: either true or false. (see below for more details).

This function unlocks the mutex m by making it unlocked/not-abandoned. It is not an error to unlock an unlocked mutex and a mutex that is owned by any task. If the monitor (or condition-variable) monitor is supplied, the current task is blocked and added to the monitor before unlocking mutex; the task can unblock at any time but no later than when an appropriate call to monitor_notify or monitor_broadcast is performed (see below), and no later than the timeout (if timeout is supplied). If there are tasks waiting to lock this mutex, the scheduler selects a task, the mutex becomes locked/owned or locked/not-owned, and the task is unblocked. Mutex_unlock returns false when the timeout is reached, otherwise it returns true.

Reactive Variables

function: rvar()
returns: an unbound reactive variable.
function: rbind(r, obj)
returns: void

Bind the object obj to the reactive variable r.

function: rget(r)
returns: the object bound to the reactive variable r.

This function will wait till the reactive variable is bound by another task. If there are no other tasks running this function will raise a "Deadlock detected" error.
