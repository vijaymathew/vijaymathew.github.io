Moving Ahead

The preceding chapter introduced some of the most useful syntactic forms in Slogan. This chapter talks about a number of additional features and programming techniques that will allow you to write more sophisticated and efficient programs.

The Function Expression

The full syntax of the function expression is given below. 

// Returns a function object.
      
function(params, ...) body;
^(params, ...) body;

The variables in params, ... are the formal parameters of the function.

The body may begin with a sequence of definitions, in which case the bindings created by the definitions are local to the body. When the function is created, the bindings of all variables occurring free within the body, excluding the formal parameters, are retained within the function. Subsequently, whenever the function is applied to a sequence of arguments, the formal parameters are bound to the arguments, the retained bindings are restored, and the body is evaluated.

let f = let (x = 10) ^(y) x + y;  
f(100); 
//=> 110
Parameter default values and annotations

Function parameters can take default values. If an argument is missing at the call site, its default value is automatically plugged-in. An example is shown below:

let f = ^(x = 10, y = 20) x + y; 
f();
//=> 30
f(30); 
//=> 50

f(30, 100);
//=> 130

Parameters that have default values become "optional" during function call. Another way to mark parameters as optional is by using the @optional annotation. All parameters marked @optional will have false as its default value unless otherwise specified.

let f = ^(x, @optional y, z) [x, y, z]; 
f(10);
//=> [10, false, false]

f(10, 20, 30); 
//=> [10, 20, 30]

let f = ^(x, @optional y, z = 100) [x, y, z]; 
f(10);
//=> [10, false, 100]

f(10, 20, 30); 
//=> [10, 20, 30]

All functions defined so far require the arguments to be passed in the same order as the parameters. The @key annotation allows us to define functions that accept arguments in any order:

let f = ^(x, @key y, z) [x, y, z]; 
f(10);
//=> [10, false, false]

f(10 , z = 30, y = 20);  
//=> [10, 20, 30]

let f = ^(x, @key y, z = 100) [x, y, z]; 
f(10);
//=> [10, false, 100]

f(10, z = 30, y = 20);  
//=> [10, 20, 30]

The @rest annotation allows to define functions that take an arbitrary number of arguments. A single variable should follow the @rest annotation. All the additional arguments are collected in a list that is bound to this variable. The following example show a function that takes two mandatory arguments and n optional arguments:

let f = ^(x, y, @rest z) [x, y, z];
f(10, 20); 
//=> [10, 20, []]
      
f(10, 20, 30, 40);
//=> [10, 20, [30, 40]]
The Function Statement

The function statement provides a shorter way to create and name functions. 

// syntax of the function statement:
function name(params, ...) body;

// syntax of the function statement with "contracts":      
function name(params, ...) where precondition -> postcondition
  body;

A "contract" for the function can be specified using the where clause. Precondition is an expression that should return true for the arguments of the function and postcondition is an expression that should return true for the result of the function. A precondition is a contract that should be satisfied by the callers of the function and a postcondition is a guarantee made by the function regarding its output.

The variable % gets bound to the result of the function in the postcondition. The postcondition is optional.

function precond(x, y) is_integer(x) && x > 10 && is_integer(y) && y > 0;

function a(x, y) where precond(x, y) -> is_integer(%)
  if (x == 20) "ok"
  else x * y;
a(1, 2);
//-> error: precondition_failed
a(19, 2);
//=> 38
a(20, 2);
//-> error: postcondition_failed

The following functions can be called to disable/enable function contracts dynamically:

disable_function_contracts()
enable_function_contracts()
// Both functions return void.
Functions returning functions
compose(f1, ...)
Returns a function that is the composition of the functions f1, ....
The returned function takes a variable number of arguments, applies the rightmost of f1, ...s
to the arguments, the next function in f1, ... (right-to-left) to the result, etc.

Examples:

let negative_quotient = compose(sub, div); 
negative_quotient(8, 3); 
//=> -8/3

compose(number_to_string, add)(10, 20, 8, 100); 
=> "138"

If compose is called with no arguments the identity function (described below) is returned.

identity(obj) 
// returns `obj`, where `obj` is any valid Slogan object.
mapfn(f)
Returns a function that takes a sequence `xs` as its argument and returns a sequence of the
results of applying `f` to each element of `xs.

Examples:

let doubler = mapfn(^(x) x * 2);
doubler([1,2,3,4,5]);
//=> [2, 4, 6, 8, 10]

let adder = mapfn(add); 
adder([1, 2, 3], [4, 5, 6]);
//=> [5, 7, 9]
partial(f, @rest args)
Returns a function `pf` that applies `f` to `args` and any more arguments passed to `pf`.

Examples:

function mult_3(a, b, c) a * b * c; 
let mult_2_with_9 = partial(mult_3, 9); 
mult_2_with_9(2, 3); 
//=> 54

let mult_with_18 = partial(mult_2_with_9, 2); 
mult_with_18(10); 
//=> 180
complement(f)
Returns a function that accept the same arguments as the function `f`, has the same effects,
if any, and returns the opposite truth value.

Recursion and tail calls

Slogan do not have special iteration constructs like the while and for loops. Iteration in Slogan is expressed more clearly and succinctly via recursion. Recursion is more general and eliminates the need for variable assignments, resulting in code that is more reliable and easier to follow.

Some recursion is essentially iteration and executes as such. When a function call is in tail position (see below) with respect to a function expression, it is considered to be a tail call, and Slogan treats it as a "goto" or jump. When a procedure tail-calls itself or calls itself indirectly through a series of tail calls, the result is tail recursion. Because tail calls are treated as jumps, tail recursion can be used for indefinite iteration in place of the more restrictive iteration constructs provided by other programming languages, without fear of overflowing any sort of recursion stack.

A call is in tail position with respect to a function expression if its value is returned directly from the function expression, i.e., if nothing is left to do after the call but to return from the function expression. For example, a call is in tail position if it is the last expression in the body of a function expression, the consequent or alternative part of an if expression in tail position, the last subexpression of an and or or expression in tail position, the last expression in the body of a let or letrec in tail position, etc. Each of the calls to f in the expressions below are tail calls, but the calls to g are not:

^() f(g()); 
^() if (g()) f() else f(); 
^() let (x = 4) f(); 
^() or(g(), f());

The letrec expression is used to define local, recursive functions. 

letrec (var = expr, ...) body;

Unlike let, the variables var... are visible not only within the body of the letrec but also within expr....

Examples:

letrec (sum = ^(xs) 
                if (is_empty(xs)) 0 
                else head(xs) + sum(tail(xs))) 
  sum([1, 2, 3, 4, 5]);
//=> 15

// We can also define mutually recursive functions using `letrec`:
letrec (is_even = ^(x) or(x == 0, is_odd(x - 1)), 
        is_odd = ^(x) and(not(x == 0), is_even(x - 1))) 
  [is_even(20), is_odd(20)]; 
//=> [true, false]

In a letrec, the expressions being bound to the variables need not be functions. The only restriction is that it must be possible to evaluate each expression without evaluating any of the variables. This restriction do not apply to function bodies because they cannot be evaluated until the functions are called in the letrec body. The following letrec expression obeys this restriction:

 letrec (f = ^() x + 2, x = 1) f(); 
//=> 3

The letrec given below will raise an exception indicating that x is unbound:

letrec (y = x + 2, x = 1) y;

When a recursive function is called in only one place outside the function, it is often clearer to use a named let expression, which take the following form:

let name (var = expr, ...) body

Named let is similar to unnamed let in all respects. In addition, the variable name is bound within the body to a function that may be called to recur; the arguments to the function become the new values for the variables var.... Here is a simple looping construct made with a named let:

let loop (xs = [1, 3, 4, 2], sum = 0) 
  if (is_empty(xs)) sum 
  else loop(tail(xs), sum + head(xs)); 
//=> 10

Generic Methods

Slogan support multiple dispatch or multimethods by which a function can be dynamically dispatched based on the run time (dynamic) type of more than one of its arguments. Functions defined for multiple dispatch are called methods. A method is a specialization of a generic function, which is introduced with the declare statement. The following example shows how to implement a generic function that can be used to join two objects of different types together to form a list:

declare generic join(a, b); 

method join(a:list, b:array) append(a, array_to_list(b)); 
method join(a:list, b:string) append(a, string_to_list(b)); 

join([1, 2, 3], #[10, 20, 30]); 
//=> [1, 2, 3, 10, 20, 30]

join([1, 2, 3], "hello"); 
=> [1, 2, 3, !h, !e, !l, !l, !o]

join([1, 2, 3], 43); 
//-> error: method_not_defined

The generic method join was defined for combinations of a list with an array or a string. The last invocation raised as error because join was not defined for the combination of a list and an integer.

The syntax of the method statement is summarized as follows:

method name(param:type, ...) body;

Types is the name of any built-in or user-defined type that can be mapped to a type checking predicate of the form is_type_name. For example, the type name list is mapped to the predicate is_list and the type name string is mapped to the predicate is_string.

The declare generic statement provides an extension that allows for shorter definitions of the generic method implementations. The following program shows how to use this feature:

declare generic join(a, b) 
  | (list, array) -> append(a, array_to_list(b))
  | (list, string) -> append(a, string_to_list(b));

The definitions can have an optional else clause where we define a method that will be called if all other argument types fail to match:

declare generic join(a, b) 
  | (list, array) -> append(a, array_to_list(b))
  | (list, string) -> append(a, string_to_list(b))
  | else -> [a, b];

join([1, 2, 3], #[10, 20, 30]); 
//=> [1, 2, 3, 10, 20, 30]

join([1, 2, 3], "hello"); 
//=> [1, 2, 3, !h, !e, !l, !l, !o]

join([1, 2, 3], 43); 
//=> [[1, 2, 3], 43]

Generic methods are used to override the built-in operators as well. As an example, let us override the addition operator (+) to work with strings. We use the @rest annotation to maintain the ability of + to take an arbitrary number of arguments.

method `+`(a:string, @rest b:string) apply(string_append, a, b);
      
"hello" + " world";
//=> "hello world"

1 + 2; 
//=> 3

`+`(1, 2, 3); 
//=> 6

`+`("a", "b", "c"); 
//=> "abc"

Continuations

While evaluating an expression, Slogan keeps track of two things: the expression to evaluate and what to do with its value. Consider the expression x < y in the following program:

if (x < y) say_hello() else say_goodbye();

The expression to evaluate is x < y and "what to do with its value" is to make a decision on which of the two functions - say_hello and say_goodbye - to call. The "what to do with the value" is called the continuation of a computation. Thus, at any point during the evaluation of an expression, there is a continuation ready to complete, or at least continue, the computation from that point.

The continuation of any expression can be captured with the function callcc. Callcc must be passed a function p of one argument. The current continuation itself is represented as a function k. This function is passed to p. If k is applied to a value from within p, it returns the value to the continuation of the callcc application. This value becomes, in essence, the value of the application of callcc. If p returns without invoking k, the value returned by the procedure becomes the value of the application of callcc. Consider these simple examples:

2 + callcc(^(k) 3 + 4); 
//=> 9

2 + callcc(^(k) k(3 + 4)); 
//=> 9

2 + callcc(^(k) { k(10); k(3 + 4) }); 
//=> 12

2 + callcc(^(k) k(3) + 4); 
//=> 5

In the first case, the continuation k is never used, allowing callcc to return 7. In the second case, the continuation is used but its effect is the same as the previous case. In the third case, the continuation is invoked two times. The first call returns the value 10 to the continuation of the application of callcc and the second use of the continuation is never reached. The fourth case is similar. Calling the continuation with the value 3 leads to the addition being ignored.

Continuations can be used to implement advanced control structures. Here is an example of using continuations to implement the return statement found in most imperative languages: 

function product(xs) 
  callcc(^(return) 
           let loop (xs = xs, p = 1) 
             if (is_empty(xs)) p 
             else if (head(xs) == 0) return(p)
             else loop(tail(xs), p * head(xs))); 

product([1, 2, 3, 4, 5]); 
//=> 120

product([1, 2, 3, 0, 4, 5]); 
//=> 6

The return continuation allows product to exit immediately upon seeing a zero, without performing the remaining multiplications.

Another use of continuations is to implement various forms of multitasking, like the coroutine implementation shown below:

function a(k) 
{ showln(1); 
  callcc(k); 
  showln(3) }; 

function b(k) 
{ showln(2); 
  callcc(k) }; 

a(b); 
//-> 1
     2
     3

Some functions related to continuations and control flow and discussed below.

dynamic_wind(in, body, out)
Returns the result of evaluating `body`. `dynamic_wind` is useful for performing
tasks that must be performed whenever control enters or leaves `body`, either normally or
by continuation application. The three arguments - `in`, `body`, and `out` - must
be functions that accept zero arguments. Before applying `body`, and each time `body`
is entered subsequently by the application of a continuation created within `body`,
the `in` function is applied. Upon normal exit from `body` and each 
time `body` is exited by the application of a continuation created outside `body`,
the `out` function is applied. Thus, it is guaranteed that `in` is invoked at
least once. In addition, if `body` ever returns, `out` is invoked at least once.
values(@rest obj)
The function `values` accepts any number of arguments and simply
passes (returns) the arguments to its continuation.
call_with_values(producer, consumer)
Returns the value returned by `consumer`.

Both `producer` and `consumer` must be functions. `call_with_values` applies
`consumer` to the values returned by invoking `producer` without arguments.

Examples:

call_with_values(^values('a, 'b), ^(x, y) [y, x]);
//=> [b a]

function a(x, y) values(y, x);
call_with_values(^a(10, 20), ^(x, y) [x, y]);
//=> [20, 10]
