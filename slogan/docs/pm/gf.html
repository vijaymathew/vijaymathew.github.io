Moving Ahead

The preceding chapter introduced the most useful primitive syntactic forms and procedures that will get you started in programming Slogan. This chapter introduces a number of additional features and programming techniques that will allow you to write more sophisticated and efficient programs.

Delayed functions and Macros

In the Introduction we met the functions and and or which are said to evaluate their arguments lazily. Let us try to understand the concept of lazy evaluation by implementing one of these functions ourselves, say and. To avoid confusing with the built-in function, we will call the version we write my_and. Here is our first attempt:

slogan> function my_and(a, b) 
          if (a && b) true 
          else false;

One limitation of my_and is that it can accept only two arguments. The built-in and function accepts an arbitrary number of arguments. We will fix this limitation later. But my_and has an even more serious problem as you can see from the following example:

slogan> my_and(false, 1/0); 
error: Divide by zero

All the arguments of my_and is evaluated before the function body is called. This leads to the "Divide by zero" error. The built-in and function do not have this problem because there is something special in the way it is defined that delays the evaluation of arguments until they are needed in the function body. In the previous example, there is no need for the function to evaluate the second argument as its return value has already being determined from the first argument. We can add lazy evaluation of arguments to my_and by adding the keyword lazy to its definition:

slogan> function lazy my_and(a, b) 
          if (a && b) true
          else false;

This definition of my_and has almost the same behavior as and when called with two arguments:

slogan> my_and(false, 1/0); 
=> false
slogan> and(false, 1/0);
=> false

We said "almost the same behavior" because and returns the last argument if all arguments are not false. But my_and will return true:

slogan> and(1, 2); 
=> 2
slogan> my_and(1, 2); 
=> true

We also need to accept 0-n number of arguments, just like the built-in and. Let us have all these problems fixed in the next version of my_and:

slogan> function lazy my_and args 
          if (is_empty(args)) true 
          else if (length(args) == 1) 
           force(head(args)) 
          else 
           if (force(head(args))) apply(my_and, tail(args)) 
           else false;

This implementation requires some explanation. First of all the parameter list is just an identifier called args. This is a shortcut for making a function accept an arbitrary number of arguments. When calling the function, all the arguments will be packaged into a single list named args. If this list is empty, i.e, no arguments were specified, the function returns true. If there is only one argument, its value is returned. If there are more than one argument, we check if the first argument is true. If it is true we recursively call the function until the list is reduced to a single argument. As the delayed arguments are packaged into a list, we have to explicitly call the force function to check their values. (The explicit call to force is not required for lazy functions that take a fixed number of arguments.)

Now we can use my_and anywhere we would use the built-in and function:

slogan> my_and();
=> true
slogan> my_and(1);
=> 1
slogan> my_and(1, false, 1/0);
=> false
Macros

Macros are constructs that are syntactically similar to functions but there is a subtle difference in the way a call to a macro is executed. When the compiler sees a function call, it inserts code to evaluate the arguments, push the values to the runtime stack and makes the processor jump to the beginning of the function's body. When the compiler sees a macro call, it invokes a program called the macro expander. The macro expander makes a copy of the macro body, replaces the parameter references with the actual argument expressions and puts the macro body in-place of the macro call itself. When control returns to the compiler, it sees the macro body instead of the macro call. It is this in-lined macro body that gets compiled to machine code.

Macros gives us a way to write language extensions that do no incur the runtime cost of a function call. Macros allows us to write syntactic extensions that conditionally evaluate their arguments, similar in effect to lazy functions but with better performance. As of now, macros have one limitation: a macro cannot make recursive calls to itself.

As an example of using macros, let us define a loop abstraction called dotimes. This is a construct that will act as a basic counting loop:

slogan> macro dotimes(n, body_fn) 
          let loop (i = 0, f = body_fn) 
           if (i < n) 
            { f(i); 
              loop(i + 1, f) };

This macro has two parameters: n must be a positive integer and body_fn must be a function that takes a single argument. Body_fn is called n times with the current count as argument. Here is an example usage:

slogan> dotimes(5, fn (i) showln(i));
-> 0
   1
   2
   3
   4

The compiler will use the macro expander to replace the call to dotimes with the following code:

let loop (i = 0, f = fn (i) showln(i)) 
 if (i < n) 
  { f(i); 
    loop(i + 1, f) };

This expanded code is what gets compiled into machine instructions. Macros are ideal for small utilities like dotimes. A large chunk of code written as a macro can lead to significant code bloat if it is called from many sites in the program. In such cases, a lazy function is a better alternative.

Recursion

The dotimes macro allows us to repeatedly perform an expression. Another way to achieve this is by writing a function that contains the expression and invoke the function more than once as part of the expression itself. This is called recursion. In simple words, recursion is the application of a function from within that function. At first recursion might seem more difficult than looping constructs like dotimes, but once mastered it provides much more expressive power.

Perhaps the simplest recursive function that we can define is the following:

slogan> function loop_forever() 
          loop_forever();

This function takes no arguments and simply calls itself immediately. If you call this function, it will hang the REPL, because it never returns. (Type Ctrl + C to terminate the REPL.)

For a recursive function to be useful, we must have some way to terminate the recursion. Most recursive functions should have at least two basic elements, a base case and a recursion step. The base case terminates the recursion, giving the value of the function for some base argument. The recursion step gives the value in terms of the value of the function applied to a different argument. In order for the recursion to terminate, the different argument must be closer to the base argument in some way.

The canonical example of a recursion is the function to find the length of a list. The base case of this function is the empty list, where the function will return the length as zero. In order to get closer to the empty list, the recursive call should be made by eliminating the first element of the list. There are two built-in functions that lets us take apart a list: the head function will return the first element and the tail function will return the rest of the elements. A nonempty list is one element longer than its tail, so the recursion step gives the value as one more than the length of the tail of the list:

slogan> function my_length(xs) 
          if (is_empty(xs)) 0 
          else 1 + my_length(tail(xs));

(To avoid confusion between the built-in function length, we have called our implementation my_length).

slogan> my_length([]); 
=> 0
slogan> my_length([10, 56, 100]); 
=> 3

Slogan allow you to trace the execution of a function to see how it operates. Type trace(name), where name is the name of a function you have defined at top level. A function can be removed from the list of traced functions by calling untrace(name). Let us try to trace the recursive calls to my_length for the argument [10, 56, 100]:

slogan> trace(my_length); 
slogan> my_length([10, 56, 100]); 
-> 
| > (my_length '(10 56 100))
| | > (my_length '(56 100))
| | | > (my_length '(100))
| | | | > (my_length '())
| | | | 0
| | | 1
| | 2
| 3          
=> 3

The indentation shows the nesting level of the recursion; the vertical lines associate applications visually with their values. Notice that on each application of length the list gets smaller until it finally reaches the empty list. The value at the empty list is 0, and each outer level adds 1 to arrive at the final value.

(Trace displays code translated to Scheme, the intermediate language used by the Slogan compiler. As Scheme is also a high-level language, it is not hard to understand the traced output.)

So far we have defined recursive functions as top-level definitions. Is it possible to create a local recursive function using let? Let us give it a try by defining a local function that sums a list of integers:

slogan> let (sum = fn (xs) 
                    if (is_empty(xs)) 0 
                    else head(xs) + sum(tail(xs))) 
         sum([1, 2, 3, 4, 5]); 

-> error: Unbound variable: sum

This results in the "Unbound variable" error because the variable sum is visible only within the body of the let expression and not within the function expression whose value is bound to sum. We can get around this problem by passing the function sum to itself as follows:

slogan> let (sum = fn (sum, xs) 
                    if (is_empty(xs) ) 0 
                    else head(xs) + sum(sum, tail(xs))) 
         sum(sum, [1, 2, 3, 4, 5]); 
=> 15

Luckily, to define local recursive functions we don't have to resort to clever solutions like this. Instead, we can use the letrec expression. The syntax of letrec is similar to let:

letrec (var = expr, ...) body;

Unlike let, the variables var... are visible not only within the body of the letrec but also within expr.... Thus, we can rewrite the expression above as follows:

slogan> letrec (sum = fn (xs) 
                       if (is_empty(xs)) 0 
                       else head(xs) + sum(tail(xs))) 
         sum([1, 2, 3, 4, 5]); 
=> 15

Using letrec we can define mutually recursive functions like the functions is_even and is_odd shown below:

slogan> letrec (is_even = fn (x) or(x == 0, is_odd(x - 1)), 
                is_odd = fn (x) and(not(x == 0), is_even(x - 1))) 
         [is_even(20), is_odd(20)]; 
=> [true, false]

In a letrec, the expressions being bound to the variables need not be functions. The only restriction is that it must be possible to evaluate each expression without evaluating any of the variables. This restriction do not applies to function bodies because they cannot be evaluated until the functions are called in the letrec body. The following letrec expression obeys this restriction:

slogan> letrec (f = fn () x + 2, 
                x = 1) 
         f(); 
=> 3

The following letrec will raise an exception indicating that x is unbound:

slogan> letrec (y = x + 2, 
                x = 1) 
         y ;

When a recursive function is called in only one place outside the function, it is often clearer to use a named let expression, which take the following form:

let name (var = expr, ...) body

Named let is similar to unnamed let in all respects. In addition, the variable name is bound within the body to a function that may be called to recur; the arguments to the function become the new values for the variables var.... Here is a simple looping construct made with a named let:

slogan> let loop (xs = [1, 3, 4, 2], sum = 0) 
         if (is_empty(xs)) sum 
         else loop(tail(xs), sum + head(xs)); 
=> 10
Tail calls

At this point, readers who are familiar with other languages that provide special iteration constructs, e.g., while or for loops, might wonder whether similar constructs are required in Slogan. Such constructs are unnecessary; iteration in Slogan is expressed more clearly and succinctly via recursion. Recursion is more general and eliminates the need for the variable assignments required by many other languages' iteration constructs, resulting in code that is more reliable and easier to follow.

Some recursion is essentially iteration and executes as such. When a function call is in tail position (see below) with respect to a function expression, it is considered to be a tail call, and Slogan treats it as a "goto" or jump. When a procedure tail-calls itself or calls itself indirectly through a series of tail calls, the result is tail recursion. Because tail calls are treated as jumps, tail recursion can be used for indefinite iteration in place of the more restrictive iteration constructs provided by other programming languages, without fear of overflowing any sort of recursion stack.

A call is in tail position with respect to a function expression if its value is returned directly from the function expression, i.e., if nothing is left to do after the call but to return from the function expression. For example, a call is in tail position if it is the last expression in the body of a function expression, the consequent or alternative part of an if expression in tail position, the last subexpression of an and or or expression in tail position, the last expression in the body of a let or letrec in tail position, etc. Each of the calls to f in the expressions below are tail calls, but the calls to g are not:

slogan> fn () f(g()); 
slogan> fn () if (g()) f() f(); 
slogan> fn () let (x = = 4) f(); 
slogan> fn () or(g(), f());

As we have seen earlier, the proper treatment of tail calls avoids the recursion stack from growing too "deep". Consider the following recursive definition of the factorial function:

slogan> function factorial(n) 
          if (n == 0) 1 
          else n * factorial(n - 1);

The recursive call to factorial is not in the tail position. An invocation of this function will lead the stack to grow n steps deep. If n is too large, this might lead to a "stack overflow" error. We can see this growing stack by tracing a call to factorial:

slogan> trace(factorial);
slogan> factorial(5);
->
| > (factorial 5)
| | > (factorial 4)
| | | > (factorial 3)
| | | | > (factorial 2)
| | | | | > (factorial 1)
| | | | | | > (factorial 0)
| | | | | | 1
| | | | | 1
| | | | 2
| | | 6
| | 24
| 120

=> 120

The recursive call can be moved to a tail position by adding one more argument to factorial which will act as an accumulator for the function's result:

slogan> function factorial(n, a) 
          if (n == 0) a 
          else factorial(n - 1, a * n);

Now if we trace factorial again, we will see that the "shape" of recursion has changed. It will now have a stack that is always at a single level:

slogan> trace(factorial);
slogan> factorial(5, 1);
->
| > (factorial 5 1)
| > (factorial 4 5)
| > (factorial 3 20)
| > (factorial 2 60)
| > (factorial 1 120)
| > (factorial 0 120)
| 120

=> 120

The function has the inconvenience that the caller has to pass the correct initial value of the accumulator. This limitation can be overcome by making the iterative function local by using a named let. The following definition of factorial preserves the original single argument interface of the function while keeping the recursive calls at the tail position:

slogan> function factorial(n) 
           let loop(n = n, a = 1) 
            if (n == 0) a 
            else loop(n - 1, a * n);

Continuations

While evaluating an expression, Slogan keeps track of two things: the expression to evaluate and what to do with its value. Consider the expression x < y in the following program:

if (x < y) say_hello() else say_goodbye();

The expression to evaluate is x < y and "what to do with its value" is to make a decision on which of the two functions - say_hello and say_goodbye - to call. The "what to do with the value" is called the continuation of a computation. Thus, at any point during the evaluation of any expression, there is a continuation ready to complete, or at least continue, the computation from that point.

The continuation of any expression can be captured with the function callcc. Callcc must be passed a function p of one argument. The current continuation itself is represented as a function k. This is function is passed to p. If k is applied to a value from within p, it returns the value to the continuation of the callcc application. This value becomes, in essence, the value of the application of callcc. If p returns without invoking k, the value returned by the procedure becomes the value of the application of callcc. Consider some simple examples:

slogan> 2 + callcc(fn (k) 3 + 4); 
=> 9
slogan> 2 + callcc(fn (k) k(3 + 4)); 
=> 9
slogan> 2 + callcc(fn (k) { k(10); k(3 + 4) }); 
=> 12
slogan> 2 + callcc(fn (k) k(3) + 4); 
=> 5

In the first case, the continuation k is never used allowing callcc to return 7. In the second case, the continuation is used but its effect is the same as the previous case. In the third case, the continuation is invoked two times. The first call returns the value 10 to the continuation of the application of callcc and the second use of the continuation is never reached. The fourth case is similar. Calling the continuation with the value 3 leads to the addition being ignored.

Continuations can be used to implement advanced control structures. Here is an example of using continuations to implement the return statement found in most imperative languages: 

slogan> function product(xs) 
          callcc(fn (return) 
                  let loop (xs = xs, p = 1) 
                   if (is_empty(xs)) p 
                   else if (head(xs) == 0) return(p)
                   else loop(tail(xs), p * head(xs))); 

slogan> product([1, 2, 3, 4, 5]); 
=> 120
slogan> product([1, 2, 3, 0, 4, 5]); 
=> 6

The return continuation allows product to exit immediately upon seeing a zero value without performing the remaining multiplications.

Another use of continuations is to implement various forms of multitasking, like the simple coroutine implementation shown below:

slogan> function a(k) 
          { showln(1); 
            callcc(k); 
            showln(3) }; 

slogan> function b(k) 
          { showln(2); 
            callcc(k) }; 

slogan> a(b); 
-> 1
   2
   3

The Function Expression

This section gives you the full syntax of the function expression. 

syntax: function (params, ...) body;
returns: a function
(To save a few keystrokes, the keyword fn can be used instead of the keyword function.)

The variables in params, ... are the formal parameters of the function.

The body may begin with a sequence of definitions, in which case the bindings created by the definitions are local to the body. When the function is created, the bindings of all variables occurring free within the body, excluding the formal parameters, are retained with the function. Subsequently, whenever the function is applied to a sequence of arguments, the formal parameters are bound to the arguments, the retained bindings are restored, and the body is evaluated.

slogan> define f = let (x = 10) fn (y) x + y;  
slogan> f(100); 
=> 110
Parameter Default Values and Annotations

Function parameters can be declared to take default values using the assignment notation. If an argument is missing at the call site, its default value will be automatically plugged-in. An example is shown below:

slogan> define f = fn (x = 10, y = 20) x + y; 
slogan> f();
=> 30
 slogan> f(30); 
=> 50
slogan> f(30, 100);
=> 130

Parameters that has default values becomes optional during function call. Another way to mark parameters as optional is by using the @optional annotation. All parameters that follows the annotation will be treated as optional. All parameters marked @optional will have false as default value unless otherwise specified:

slogan> define f = fn (x, @optional y, z) [x, y, z]; 
slogan> f(10);
=> [10, false, false]
slogan> f(10, 20, 30); 
=> [10, 20, 30]

slogan> define f = fn (x, @optional y, z = 100) [x, y, z]; 
slogan> f(10);
=> [10, false, 100]
slogan> f(10, 20, 30); 
=> [10, 20, 30]

All the functions defined so far requires the arguments to be passed in the same order as the parameters. The @key annotation allows us to define functions that accept arguments in any order:

slogan> define f = fn (x, @key y, z) [x, y, z]; 
slogan> f(10);
=> [10, false, false]
slogan> f(10 , z = 30, y = 20);  
=> [10, 20, 30]

slogan> define f = fn (x, @key y, z = 100) [x, y, z]; 
slogan> f(10);
=> [10, false, 100]
slogan> f(10, z = 30, y = 20);  
=> [10, 20, 30]

The @rest annotation allows us to define functions that take an arbitrary number of arguments. A single variable should follow the @rest annotation. All the additional arguments are collected in a list that is bound to this variable. The following example shows a function that takes two mandatory arguments and n optional arguments:

slogan> define f = fn (x, y, @rest z) [x, y, z];
slogan> f(10, 20); 
=> [10, 20, []]
slogan> f(10, 20, 30, 40);
=> [10, 20, [30, 40]]
Function Statement

The function (or fn) statement combines the function and define expressions to give us a shorter way to create named functions. 

syntax: function name (params, ...) body;
syntax: function name params body;

The second syntax defines a function that takes 0 - n arguments. All the arguments are packaged into a list (params) and passed to the function. Both functions defined below will appear to the caller as having similar definitions:

slogan> function f(@rest a) a; 
slogan> function g a a;

slogan> f(1, 2, 3);
=> [1, 2, 3]
slogan> f();
=> []
slogan> g(1, 2, 3);
=> [1, 2, 3]
slogan> g();
=> []
Composing Functions
function: compose(f1, ...) 
returns: a function that is the composition of the functions f1, ....

The returned function takes a variable number of arguments, applies the rightmost of f1, ...s to the arguments, the next function in f1, ... (right-to-left) to the result, etc.

slogan> define negative_quotient = compose(sub, div); 
slogan> negative_quotient(8, 3); 
=> -8/3
slogan> compose(number_to_string, add)(10, 20, 8, 100); 
=> "138"

If compose is called with no arguments the identity function (described below) is returned.

function: identity(obj) 
returns: obj.
