Functional Concurrency

Consider a simple concurrent program using reactive variables:

slogan> define x = ?; 
slogan> define y = ?; 

slogan> ?x = 100; 
slogan> task({ ?y = ?x }); 
slogan> ?y; 
=> 100

The line "task({ ?y = ?x });" is the concurrent part of the program. It runs the assignment statement in a new task separate from the main task of the REPL. It is also an example of functional concurrency because for a given value of x it always leads to the same output despite the order in which x is bound. This is made clear in the following program:

slogan> define x = ?; 
slogan> define y = ?; 

slogan> task({ ?y = ?x }); 
slogan> ?x = 100; 
slogan> ?y; 
=> 100

Let us write a concurrent version of the map function. Here we will see a more extensive use of reactive variables for coordination and communication between tasks:

slogan> function cmap(f, xs) 
          match (xs) 
            []: [], 
            [h t]: let (x = ?) 
                    { task({ ?x = f(?h) }); 
                      [?x cmap(f, ?t)] };

The function is called cmap for "concurrent map". It calls the function f for each element in the list xs in a dedicated task. It also treats the head and tail of xs as reactive variables, allowing it to block and wait for values to be bound by another task that acts as some kind of "value producer". Here is our concurrent map function in action: 

slogan> define xs = ?;
slogan> define ys = ?;
slogan> define result = ?;

slogan> task({ ?result = cmap(sqrt, ?xs)});
slogan> ?xs = [10, 20, ys];

We call cmap in a new task and pass it the reactive variable xs. This task will block until xs is bound to the list [10, 20, ys]. This will trigger cmap to compute the square roots of 10 and 20 and then block on ys. To run cmap to completion and view the result, now we have to bind ys:

slogan> ?ys = 30; 
slogan> ?result;
=> [3.1622776601683795, 4.47213595499958, 5.477225575051661]

The result is the same as what the sequential map function will return. But with the concurrent map function, we are able to generate the output incrementally for an incrementally given input.

Streams

A stream is a potentially unbounded list of messages, i.e., it is a list whose tail is an unbound reactive variable. Sending a message is done by extending the stream by one element: bind the tail to a list pair containing the message and a new unbound tail. Receiving a message is reading a stream element. Streams is a useful tool in programming communicating concurrent tasks in a functional setting. The task generating the messages is called a producer and the task consuming the messages is known as a consumer.

Producer/consumer

Here is the most basic producer/consumer function pair we can write:

slogan> function producer(xs, limit) 
          let loop(n = 0, r = xs) 
           if (n == limit) ?r = [] 
           else let (nr = ?) 
                 { ?r = [n nr]; 
                   loop(n + 1, nr) } ; 

slogan> function consumer(xs, a) 
          match (xs) 
           []: a, 
           [x xr]: consumer(?xr, ?x + a);

The producer function generates a stream of integers until limit is reached. The reactive variable xs is used to maintain the unbound tail of the stream which a consumer task can use to wait for more values.

The consumer function reads from a stream and accumulates the sum of integers read into a. The following program shows how to use the producer/consumer pair:

slogan> define xs = ?; 
slogan> define result = ?; 
slogan> task({ ?result = consumer(?xs, 0) }); 
slogan> producer(xs, 150000); 
slogan> ?result;
=> 11249925000
Demand-driven flow control

Sometimes the producer will generate more elements than the consumer can process. This can bog down system resources. One way to solve this problem is to limit the rate at which the producer generates new elements, so that some global condition is satisfied. This is called flow control. It requires that some information be sent back from the consumer to the producer. The simplest flow control is to let the producer to generate an element only when the consumer demands for it. This is called demand-driven flow control. (The previous technique, where the producer generates an element whenever it likes, can be called supply-driven). Here is how to program a demand-driven producer/consumer pair:

slogan> function producer(xs, n) 
          if (not(is_empty(?xs))) 
           match (?xs)
            [x xr]: 
              { ?x = n; 
                producer(xr, n + 1) }; 

slogan> function consumer(xs, a, limit) 
          let (xr = ?) 
           { ?xs = [?xr]; 
             let (x = head(?xs)) 
              if (?x == limit) { ?xr = []; a } 
	      else consumer(xr, ?x + a, limit) } ;

The consumer notifies the producer by binding a reactive variable to a pair whose head and tail are also unbound reactive variables. The producer waits for the pair to get bound and sends back the next value in the stream by binding it to the head. The producer terminates when it receives an empty list instead of a pair. As you might have noticed, the is_done function to decide when to stop generating the stream is passed to the consumer because it drives the whole process. We can re-write the previous "sum integer stream" example using the new producer/consumer pair as:

slogan> define xs = ?;
slogan> define result = ?;
slogan> task({ ?result = consumer(xs, 0, 150000) }); 
slogan> producer(xs, 0); 
slogan> ?result;
=> 11249925000

We have seen two techniques for managing stream communication, namely supply-driven and demand-driven. Both techniques have problems. supply-driven execution leads to an explosion in resource usage. But demand-driven execution also has a serious problem. It leads to a strong reduction in throughput. By throughput we mean the number of messages that can be sent per unit of time. If the consumer requests a message, then the producer has to calculate it, and meanwhile the consumer waits. If the producer were allowed to get ahead of the consumer, then the consumer would not have to wait.

An optimum producer/consumer pair can be designed using a bounded buffer. A bounded buffer stores elements up to a maximum number, say n. The producer is allowed to get ahead of the consumer, but only until the buffer is full. This limits the extra resource usage to n elements. The consumer can take elements from the buffer immediately without waiting. This keeps throughput high. When the buffer has less than n elements, the producer is allowed to produce more elements, until the buffer is full.
