Functional Concurrency

Consider a simple concurrent program using reactive variables:

slogan> define x = ?; 
slogan> define y = ?; 

slogan> ?x = 100; 
slogan> task({ ?y = ?x }); 
slogan> ?y; 
=> 100

The line "task({ ?y = ?x });" is the concurrent part of the program. It runs the assignment statement in a new task separate from the main task of the REPL. It is also an example of functional concurrency because for a given value of x it always leads to the same output despite the order in which x is bound. This is made clear in the following program:

slogan> define x = ?; 
slogan> define y = ?; 

slogan> task({ ?y = ?x }); 
slogan> ?x = 100; 
slogan> ?y; 
=> 100

Let us write a concurrent version of the map function. Here we will see a more extensive use of reactive variables for coordination and communication between tasks:

slogan> function cmap(f, xs) 
          match (xs) 
            []: [], 
            [h t]: let (x = ?) 
                    { task({ ?x = f(?h) }); 
                      [?x cmap(f, ?t)] };

The function is called cmap for "concurrent map". It calls the function f for each element in the list xs in a dedicated task. It also treats the head and tail of xs as reactive variables, allowing it to block and wait for values to be bound by another task that acts as some kind of "value producer". Here is our concurrent map function in action: 

slogan> define xs = ?;
slogan> define ys = ?;
slogan> define result = ?;

slogan> task({ ?result = cmap(sqrt, ?xs)});
slogan> ?xs = [10, 20, ys];

We call cmap in a new task and pass it the reactive variable xs. This task will block until xs is bound to the list [10, 20, ys]. This will trigger cmap to compute the square roots of 10 and 20 and then block on ys. To run cmap to completion and view the result, now we have to bind ys:

slogan> ?ys = 30; 
slogan> ?result;
=> [3.1622776601683795, 4.47213595499958, 5.477225575051661]

The result is the same as what the sequential map function will return. But with the concurrent map function, we are able to generate the output incrementally for an incrementally given input.

Streams

A stream is a potentially unbounded list of messages, i.e., it is a list whose tail is an unbound reactive variable. Sending a message is done by extending the stream by one element: bind the tail to a list pair containing the message and a new unbound tail. Receiving a message is reading a stream element. Streams is a useful tool in programming communicating concurrent tasks in a functional setting. The task generating the messages is called a producer and the task consuming the messages is known as a consumer.

Producer/consumer

Here is the most basic producer/consumer function pair we can write:

slogan> function producer(xs, n, is_done, next) 
          let loop (n = n, r = xs) 
           if (is_done(n)) ?r = [n] 
           else let (nr = ?) 
                 { ?r = [n nr]; 
                   loop(next(n), nr) }; 

slogan> function consumer(xs, f, a) 
           match (xs) 
            []: a, 
            [x xr]: consumer(?xr, f, f(a, ?x));

The producer function generates a stream of values until is_done returns true. Each value starting at n is calculated using the function next. The reactive variable xs is used to maintain the unbound tail of the stream which a consumer task can use to wait for more values.

The consumer function reads from a stream, applies the function f to each element and accumulates a result into a. The following program shows how to use the producer/consumer pair to write function that finds the sum of numbers read from a stream:

slogan> define xs = ?; 
slogan> define result = ?; 
slogan> task({ ?result = consumer(?xs, add, 0) }); 
slogan> producer(xs, 150000, is_zero, fn(n) n - 1); 
slogan> ?result;
=> 11250075000
