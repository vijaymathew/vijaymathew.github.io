Shared-State Concurrency

In the shared-state model, concurrent tasks communicate by accessing and mutating shared variables. Shared-state concurrency is harder to program than functional concurrency using reactive variables. The difficulty arises out of the fact that the tasks can update or read the shared state in unpredictable orders. This happens because task scheduling is non-deterministic. It is possible to write algorithms in this model and prove their correctness by reasoning on all possible read/write interleavings. But for large programs, this technique rapidly becomes impractical. It is unwieldy and interleavings are easy to overlook.

If shared-state concurrency is difficult to program, why not just stick with functional concurrency? The problem with the functional concurrent model is that tasks must communicate in "lockstep". Two task communicating with a third task cannot execute independently; they must coordinate with each other. This is a consequence of the fact that functional programs are deterministic.

We would like to allow two threads to be completely independent and yet communicate with the same third thread. For example, we would like tasks to independently increment a shared counter. To express this, we have to leave the realm of pure functional programming. This is because two independent entities communicating with a third introduce an observable non-determinism. A simple way to solve the problem is to add explicit state to the model. But as we have already seen, reasoning directly with state and concurrency is difficult. One way to get around this problem is to make all state local to tasks and use only message-passing for coordination. Another option is to perform atomic actions on shared-state. This is the main topic of this chapter.

Mutexes

A mutex is an object that can be used to prevent a shared resource from being accessed by more than one task at the same time. A mutex can be in one of two states - locked or unlocked. An attempt to lock a mutex only succeeds if the mutex is in an unlocked state, otherwise the current task will wait. The code that is executed within the mutex lock and unlock operations is known as the critical region.

The following program shows how tasks can use a mutex object to coordinate reads and writes to a shared variable:

slogan> define m = mutex(); 
slogan> define count = 0;
slogan> function a() task({ mutex_lock(m); count = count + 1; mutex_unlock(m) }); 
slogan> function b() task({ mutex_lock(m); count = count + 1; mutex_unlock(m) }); 
slogan> a();
slogan> b();

The program above spawns two tasks that modify the value of count. Both of them lock the mutex m before performing this action. This makes sure that both tasks do not try to write to count at the same time. No matter which task writes first, count always ends up with the value of 2.

What happens if one of the tasks raises an exception after locking the mutex but before unlocking it? The second task waiting to lock the mutex will be stuck forever. This situation can be prevented by putting the critical region in the body of a try and always calling the unlock operation in the finally clause. This idea is so useful that we can express it as a macro:

slogan> macro lock(mtx, expr) 
         { mutex_lock(mtx);
           try expr 
           finally mutex_unlock(mtx) };

A macro is a pure compile-time artifact. When you call a macro, the compiler expands its body into the call-site. To make this clear, let us re-write the function that updates count using the new macro:

slogan> function a() task(lock(m, count = count + 1));

When it sees the call to lock the compiler runs a macro expander which will replace the expression lock(m, count = count + 1) with the following code:

{ mutex_lock(m);
  try count = count + 1 
  finally mutex_unlock(m) }

By writing a macro we saved some keystrokes without incurring any runtime overhead.

Reentrant mutexes

The default mutex implementation in Slogan is not reentrant. An attempt to lock a mutex that is locked implies that the current task waits even if the mutex is owned by it. This can lead to a deadlock if no other task unlocks the mutex. The following program demonstrates this behavior:

slogan> define m = mutex(); 
slogan> task({ showln("locking..."); 
               mutex_lock(m); 
               showln("locked."); 
               showln("locking again..."); 
               mutex_lock(m); 
               showln("locked again." ); 
               mutex_unlock(m); 
               mutex_unlock(m) }); 

-> slogan> locking...
  locked.
  locking again...

The task will hang trying to lock the mutex the second time. Sometimes we would want to allow a task to lock the same mutex multiple times. We will implement an extended mutex that has this reentrant behavior:

slogan> record rmutex(parent_task, mtx = mutex(), lock_count = 0); 

slogan> function rmutex_lock(rmtx) 
          if (not(rmutex_parent_task(rmtx) == current_task())) 
           { mutex_lock(rmutex_mtx(rmtx)); 
             rmutex_set_parent_task(rmtx, current_task()) }
          else rmutex_set_lock_count(rmtx, rmutex_lock_count(rmtx) + 1);

slogan> function rmutex_unlock(rmtx) 
          if (rmutex_parent_task(rmtx) == current_task())
           if (rmutex_lock_count(rmtx) == 0)
            { rmutex_set_parent_task(rmtx, false); 
              mutex_unlock(rmutex_mtx(rmtx)) }
           else rmutex_set_lock_count(rmtx, rmutex_lock_count(rmtx) - 1);

The reentrent-mutex (rmutex) record keeps track of the task that currently holds the lock on its mutex. If that task tries to lock the mutex again, it's just let through. It also maintains the number of times a task has locked a mutex to match the calls to rmutex_unlock.

Monitors

Mutexes are an important tool for building concurrent abstractions in a stateful model, but they are not sufficient. For example, consider the simple case of a bounded buffer. Consider a task trying to put an element into a full buffer. What if we want the task to wait until the buffer is not full, and then continue? This cannot be done just by using mutexes. We need a mechanism that allows a task taking elements from the buffer to notify the waiting task that the buffer is no longer full. The same mechanism will allow the task putting elements to notify a task blocked on an empty buffer.

Monitors gives us such a mechanism by adding a "wait" and "notify" operations to the mutex lock and unlock functions. Calling mutex_unlock with a monitor as its second argument will add the current task to the list of tasks waiting for the monitor to be notified. The monitor_notify function will notify one other task waiting on the monitor and monitor_broadcast will notify all tasks waiting on it.

The following program shows the implementation of a bounded buffer using mutexes and monitors:

slogan> function make_bounded_buffer(size) 
          let (buf = make_array(size), n = 0, mtx = mutex(), m = monitor()) 
           letrec (put = fn (x) 
                       { mutex_lock(mtx); 
                         if (n >= size) { mutex_unlock(mtx, m); put(x) }
                         else { array_set(buf, n, x); 
                                n = n + 1; 
                                monitor_broadcast(m); 
                                mutex_unlock(mtx) } }, 
                get = fn () 
                       { mutex_lock(mtx); 
                         if (n == 0) { mutex_unlock(mtx, m); get() }
                         else { n = n - 1; 
                                let (x = array_at(buf, n)) 
                                 { monitor_broadcast(m); 
                                   mutex_unlock(mtx); 
                                   x } } }) 
            fn (msg) 
             match (msg) 
              !put: put, 
              !get: get;

Let us try out an instance of the bounded buffer by assigning two tasks to it: one fills up the buffer while the other reads it:

slogan> define b = make_bounded_buffer(10); 
slogan> task({ showln(b.get()); showln(b.get()) }); 
slogan> task({ b.put(100); b.put(200); b.put(300) }); 
-> 100
   200

How to choose a concurrency model

In this tutorial, we have studied three approaches to concurrent programming - functional, message-passing and shared-state. How do we decide which approach to use when writing a concurrent program? Here are a few rules of thumb:

  • If using concurrency does not simplify the architecture of the program, avoid it. Stick with a sequential model.
  • If you absolutely need concurrency, modularize your program in such a way that its concurrency aspects are concentrated in as few places as possible. Most of the time, large parts of the program can be sequential.
  • If your program does not have any observable non-determinism, such as independent clients interacting with a server, then stick with functional concurrency using reactive variables.
  • If you absolutely need both state and concurrency, then use either the message-passing or the shared-state approach. The message-passing approach is often the best for programs that consist of autonomous "agents" that communicate with each other. The shared-state approach is often the best for programs that deal with a global repository of data ("database") that should be accessed and updated concurrently. Both approaches can be used together for different parts of the same program.
