Message-Passing Concurrency

Message passing is a programming style in which a program consists of independent entities that interact by sending each other messages asynchronously, i.e., without waiting for a reply. It lends itself well to building highly reliable systems. Since the message-passing entities are independent, if one fails the others can continue executing. In a properly designed system, the others reorganize themselves to continue providing a service.

A Slogan task has an associated "mail box" or "message queue" where it can receive messages send by other tasks. Messaging between tasks is accomplished by two functions: task_send and task_receive. Let us get familiar with these functions by writing a simple "game". There are three players standing in a circle, tossing a ball among themselves. When a player catches the ball, he or she picks one of the other two randomly to pass-on the ball. The players are represented by tasks. Ball catching and throwing takes place through message passing. The ball is "thrown" to a player by sending that task a message in the format - !ball : name - where name is the name of the pitcher task. A name is assigned to each task by passing it as an extra argument to the task function. The task that "catches" the ball randomly selects a player and re-throws the ball to it. To prevent the players from furiously throwing balls at each other, we have also introduced a delay of one second between each throw. Let us put all this into a single function:

slogan> function player(others) 
          { match (task_receive()) 
             !ball : p -> 
              { showln(task_name(current_task()), " > ", p); 
                task_sleep(1); 
                task_send(at(random_integer(length(others)), others), 
                          !ball : task_name(current_task())); 
                player(others) }};

A game with three players can be started as follows:

slogan> let (p1 = false, p2 = false, p3 = false) 
         { p1 = task(player([p2, p3]), name = "player1"); 
           p2 = task(player([p1, p3]), name = "player2"); 
           p3 = task(player([p1, p2]), name = "player3"); 
           task_send(p1, !ball : "referee") };

-> player1 > referee
   player3 > player1
   player1 > player3
   player3 > player1
   player2 > player3
   player1 > player2
   ...

Protocols

A protocol defines a sequence of messages that needs to be send between two parties for coordination. In the previous example we saw a simple protocol where one party waits for a message to come and then forwards that message to some other party. In this section we study other interesting protocols.

RMI

RMI (Remote Method Invocation) allows an object to call the method in another in a different operating system process. The other process could be on the same machine or on another machine connected by network. Here we present RMI in its simplest form. Instead of processes we have Slogan tasks and a "method" is what a task does when it receives a message. In the synchronous version of RMI, the client sends a request to the server and waits for the reply. First let us implement the server: 

slogan> function server() 
          { match (task_receive()) 
             [!calc, x, y] -> ?y = x * x + 2.0 * x + 2.0; 
            server() };

The server binds the result of its calculation to the reactive variable y. This variable is initialized by the client which then waits on it. The following function defines a client for our synchronous RMI server:

slogan> function client(s) 
          let (y1 = ?, y2 = ?) 
           { task_send(s, [!calc, 10.0, y1]); 
             showln(?y1); 
             task_send(s, [!calc, 20.0, y2]); 
             showln(?y2) };

The client is started by passing it a reference to its server:

slogan> client(task(server())); 
-> 122.
   442.

The RMI server handles one client at a time. This approach is simple to program with and reason about. We can also allow multiple calls from different clients to be processed concurrently. This is done by starting a new task at the server side to accept and process a client request. The server no longer serves requests sequentially. This is much harder to program with, especially if the server has internal state data. Without bothering about such complexities, let us attempt a small variation in the client:

slogan> function client(s) 
          let (y1 = ?, y2 = ?) 
           { task_send(s, [!calc, 10.0, y1]); 
             task_send(s, [!calc, 20.0, y2]); 
             showln(?y1 + ?y2) };

The updated client sends the second request without waiting for the first reply. This is an example of asynchronous RMI. Note that no change in the server is required. It still receives messages one by one and executes them sequentially. Requests are handled by the server in the same order as they are sent and the replies arrive in that order as well.

RMI with callback

In certain cases a server may need additional information from the client for full-filling the original request. In such situations the server does a callback to the client. Let us look at a new version of the previous server function where the server needs an additional parameter called delta which is known only to the client:

slogan> function server() 
          { match (task_receive()) 
             [!calc, client, x, y] -> 
               let (d = ?) 
                { task_send(client, [!delta, d]); 
                  x = x + ?d; 
                  ?y = x * x + 2.0 * x + 2.0; }; 
            server() };

The request contains the reference to the task in which the client is running. Server uses this reference to send a request back to the client to retrieve the delta. Now we should update the client so that it can handle this callback:

slogan> function client(s) 
          let (y = ?) 
           { task_send(s, [!calc, current_task(), 10.0, y]); 
             match (task_receive()) [!delta, d] -> ?d = random_real(); 
             showln(?y) };

The client sends the request and immediately waits for the delta request from the server. Once the delta is sent, it again waits for the result to be bound to y. As the delta is a randomly generated real number between 0 and 1, the result will change with each invocation:

slogan> client(task(server())); 
-> 122.30518463091119
slogan> client(task(server())); 
-> 140.05584623432674
RMI with function continuation

After sending the delta the client has to wait for the server to finish computation. A server doing a complex computation may take an unspecified amount of time to return a result. What if we do not want the client to block on the result? One solution is to move the part of code that uses the result into a task of its own. For instance, the last line of the client function can be modified as: task(showln(?y)). Another solution is to use a function continuation. This involves wrapping up the part of code that uses the result into a function and send that to the server along with the result. The server, instead of binding the result to a reactive variable, calls this function with the result as its argument. Here is a basic client/server pair using function continuation:

slogan> function server() 
          { match (task_receive()) 
             [!calc, x, cont] -> cont(x * x + 2.0 * x + 2.0); 
            server() }; 

slogan> function client(s, d) 
          let (cont = fn(y) showln(y * d)) 
           task_send(s, [!calc, 10.0, cont]); 

slogan> client(task(server()), 100); 
-> 12200

In this example, instead of a reactive variable, the client passes the function "cont" to the server. This function contains a state "d" that is local to each client. Though this function is invoked by the server, it represents the rest of the work that the client want to do with the result.
