Programming with State

The programs that we studied in this tutorial were mostly functional, i.e, they depend only on their inputs or parameters to effect a computation. A program can also have an internal parameter or state which gives the program a "long-term memory". Without state a program can have only "short-term memory", one that exists during a particular invocation of the program.

This chapter introduces program design with explicit state along its advantages and limitations.

Defining state

State can be defined as a sequence of values in time that contains the intermediate results of a desired computation. As per this definition, we can maintain state without any support from the language we are programming in. The state can exist in the programmer's mind. This is known as implicit or declarative state. Consider the following function that sums a list of numbers:

slogan> function sum_list(xs, a) 
          match (xs) 
           []: a, 
           [x xr]: sum_list(xr, a + x); 

slogan> sum_list([1, 2, 3, 4], 0);
=> 10

To calculate the sum of a list, the function has to call itself many times. The example call above, leads to the following sequence of arguments being passed to the recursive calls: 

[1, 2, 3, 4], 0
[2, 3, 4], 1
[3, 4], 3
[4], 6
[], 10

This sequence is a state. When looked at in this way, sum_list calculates with state. Yet neither the program nor the language runtime "knows" this. The state is completely in the mind of the programmer.

Sometimes it is useful for a function to have a state that lives across function calls and that is hidden from the callers. For example, we can extend sum_list to count how many times it is called. We can thread the count across callers by making it an extra parameter to the function. But this will give the function an awkward interface and burden the users by making them keep track of an extension they need not be aware of. To do it without additional arguments we need an explicit state.

An explicit state in a function is a state whose lifetime extends over more than one function call without being present in the function's arguments.

Let us update the sum_list function with a variable to keep track of the number of times it is called:

slogan> define sum_count = false;
slogan> define sum_list = let (count = 0) 
                           { sum_count = fn () count; 
                             fn (xs, a) 
                              { count = count + 1;
                                match (xs) 
                                 []: a, 
                                 [x xr]: sum_list(xr, a + x) } };

In the let expression that initializes count, we create two functions. The first one just returns the current value of count. This function is bound to the name sum_count. The next function is the same as sum_list, but it also increments count by 1 each time it is called. This function is bound to the symbol sum_list. Let us try out these functions:

slogan> sum_list([1, 2, 3, 4], 0); 
=> 10
slogan> sum_count();
=> 5

Count is not visible to code outside the let expression. So it becomes the private explicit state of these two functions. The ability to encapsulate explicit state improves the modularity of data abstractions tremendously. The access to the state is limited according to the operations of the data abstraction. This idea is at the heart of object-oriented programming.
