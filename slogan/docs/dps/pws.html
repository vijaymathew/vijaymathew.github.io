Programming with State

The programs that we have seen so far were mostly functional, i.e, they depend only on their inputs or parameters to effect a computation. A program can also have an internal parameter or state which gives it a "long-term memory". Without state a program can have only "short-term memory", one that exists during a particular invocation of the program.

This chapter introduces program design with explicit state and tries to figure out its advantages and limitations.

Defining state

State can be defined as a sequence of values in time that contains the intermediate results of a desired computation. As per this definition, we can maintain state without any support from the language we are programming in. The state can exist in the programmer's mind. This is known as implicit or declarative state. Consider the following function that sums a list of numbers:

slogan> function sum_list(xs, a) 
          match (xs) 
           [] -> a, 
           x : xr -> sum_list(xr, a + x); 

slogan> sum_list([1, 2, 3, 4], 0);
=> 10

To calculate the sum of a list, the function has to call itself many times. The example call above, leads to the following sequence of arguments being passed to the recursive calls: 

[1, 2, 3, 4], 0
[2, 3, 4], 1
[3, 4], 3
[4], 6
[], 10

This sequence is a state. When looked at in this way, sum_list calculates with state. Yet neither the program nor the language runtime "knows" this. The state is completely in the mind of the programmer.

Sometimes it is useful for a function to have a state that lives across function calls and that is hidden from the callers. For example, we can extend sum_list to count how many times it is called. We can thread the count across callers by making it an extra parameter to the function. But this will give the function an awkward interface and burden the users by making them keep track of an extension they need not be aware of. To do it without additional arguments we need an explicit state.

An explicit state in a function is a state whose lifetime extends over more than one function call without being present in the function's arguments.

Let us update the sum_list function with a variable to keep track of the number of times it is called:

slogan> define sum_count = false;
slogan> define sum_list = let (count = 0) 
                           { sum_count = fn () count; 
                             fn (xs, a) 
                              { count = count + 1;
                                match (xs) 
                                 [] -> a, 
                                 x : xr -> sum_list(xr, a + x) }};

In the let expression that initializes count, we create two functions. The first one returns the current value of count. This function is bound to the name sum_count. The next function is the same as sum_list, but it also increments count by 1 each time it is called. This function is bound to the symbol sum_list. Let us try out these functions:

slogan> sum_list([1, 2, 3, 4], 0); 
=> 10
slogan> sum_count();
=> 5

Count is not visible to code outside the let expression. So it becomes the private explicit state of these two functions. The ability to encapsulate explicit state improves the modularity of data abstractions tremendously. The access to the state is limited according to the operations of the data abstraction. This idea is at the heart of object-oriented programming.

State and system design

A "system" can be thought of as having two parts: a specification and an implementation. The specification defines how the system should behave. It is like a "contract" that the system makes with the outside world that interacts with it. We say a system is correct if the actual behavior of the system is in accord with the contract. If the system behaves differently, then we say it fails.

The specification defines how the rest of the world interacts with the system, as seen from the outside. The implementation is how the system is constructed, as seen from the inside. To use a system, all we need to know is its specification. This means that it is possible to build a system as a concentric series of layers. One can proceed step by step, building layer upon layer. At each layer, build an implementation that takes the next lower specification and provides the next higher one. It is not necessary to understand everything at once. This separation of specification and implementation is an important principle in system building and is known as the principle of abstraction.

A system should have the following properties to best support the principle of abstraction:

  • Encapsulation. It should be possible to hide the internals of a part.
  • Compositionality. It should be possible to combine parts to make a new part.
  • Instantiation/invocation. It should be possible to create many instances of a part based on a single definition. These instances "plug" themselves into their environment (the rest of the system in which they will live) when they are created.

Slogan has features that enables you to design systems with these properties. Lexical scoping supports encapsulation and higher-order programming supports instantiation. These properties do not require state; they can be used in functional programming as well. For example, it is possible to use encapsulation in declarative programs without state. On the other hand, it is also possible to use state without encapsulation, by defining the state globally so all components have free access to it.

Encapsulation and explicit state are most useful when used together. Adding state to functional programming makes reasoning about the program much harder, because the program's behavior depends on the state. For example, a function can do a side effect, i.e., it modifies state that is visible to the rest of the program. Side effects make reasoning about the program extremely difficult. Bringing in encapsulation does much to make reasoning tractable again. This is because stateful systems can be designed so that a well-defined property, called an invariant, is always true when viewed from the outside. This makes reasoning about the system independent of reasoning about its environment. This partly gives us back one of the properties that makes functional programming so attractive.

Note that an invariant does not guarantee that the component is making progress toward some goal. This means that even with invariants, programming with state is not quite as simple as functional programming. A good rule of thumb for complex systems is to keep as many components as possible functional. State should not be spread out over many components. It should be concentrated in just a few carefully selected components.

Component-based programming

The three properties of encapsulation, compositionality, and instantiation define component-based programming. A component specifies a program fragment with an inside and an outside, i.e., with a well-defined interface. The inside is hidden from the outside, except for what the interface permits. Components can be combined to make new components. Components can be instantiated, making a new instance that is linked into its environment. In this tutorial we have already encountered various forms or components like closures and message-passing tasks.

In component-based programming, the natural way to extend a component is by using composition: build a new component that contains the original one. The new component offers a new functionality and uses the old component to implement the functionality. If we add one more concept called inheritance to component-based programming, we get Object-Oriented Programming.

Stateful functional programming

In general, a stateful program is no longer functional, since running the program several times with the same inputs can give different outputs depending on the internal state. It is possible, though, to write stateful programs that behave as if they were functional, i.e., when observed from outside, it always behaves functionally.

Let us see an interesting example. We define the list reverse function using state:

slogan> function reverse(xs) 
          let (result = []) 
           let loop (xs = xs) 
            if (is_empty(xs)) result 
            else { result = head(xs) : result; 
                   loop(tail(xs)) };

Since the state is encapsulated inside the function, there is no way to tell the difference between this implementation and a functional implementation. It is often possible to take a functional program and convert it to a stateful program with the same behavior by replacing the implicit state with an explicit state. Similarly, any stateful implementation can be converted to a functional implementation as well.

Stateful Data Structures

In the chapter Functional Data Structures, we implemented a functional queue. In this section we implement a related data structure called the stack. A stack is a LIFO (Last-In-First-Out) data structure with two operations - push and pop. Push adds an element to the stack and pop removes and returns the last element added. The stack we are going to implement is stateful. It keeps its elements in an array and remembers the index of the last element inserted in a variable. (An array maps integer indexes to objects. Unlike a list, an array is of fixed size and can do faster lookups by the index of the element.) A stack instance has to be initialized with a maximum size. An attempt to add elements beyond this size will raise an exception. Here is the complete implementation of stack:

slogan> function make_stack(size) 
          letseq (elems = make_array(size), 
                  index = 0, 
                  push = fn (elem) 
                          if (index >= size) error("stack overflow!") 
                          else { array_set(elems, index, elem); 
                                 index = index + 1 }, 
                  pop = fn () 
                         if (index == 0) error("stack underflow!") 
                         else let (elem = array_at(elems, index - 1)) 
                               { index = index - 1; 
                                 elem }) 
           fn (msg) 
            match (msg) 
             !push -> push, 
             !pop -> pop;

The state of a stack consists of the array elems and the integer index. These variables are manipulated by the functions push and pop. (As these functions references the variables defined before them, we have used letseq instead of let.) The function make_stack is a constructor. Each time it is called it returns a function that takes a symbolic message as argument. This function is a closure that has references to all the variables bound in the letseq. It does a case analysis of the symbolic argument and returns an appropriate function to the caller. Let us see how to create and use a stack object:

slogan> define s = make_stack(3); 
slogan> s(!push)(10); 
slogan> s(!push)(20);
slogan> s(!pop)();
=> 20
slogan> s(!pop)();
=> 10

We can also use the dot-notation to call functions that accepts a single symbolic argument. This will look more familiar to those who have used languages like Java:

slogan> define s = make_stack(3); 
slogan> s.push(10); 
slogan> s.push(20);
slogan> s.pop();
=> 20
slogan> s.pop();
=> 10

An example: Generating random numbers

We have used the random number generators random_integer and random_real. These functions that allow the computer to "throw dice" has many useful applications in programs like games and simulations. How do we implement our own random number generators? A first approach would be to use some source of randomness in the computer itself - like the thread scheduler or electronic circuits that generate noise. This will give us true randomness but has the problem that they don't have a useful probability distribution. A second and more serious problem: the randomness cannot be reproduced except by storing the random numbers and replaying them. It might seem strange to ask for reproducibility from a source of randomness, but it is perfectly reasonable. For example, the randomness might be input to a simulator. We would like to vary some parameter in the simulator such that any variation in the simulator depends only on the parameter, and not on any variation in the random numbers.

A better solution is to calculate random numbers. For this we need a good algorithm that can generate a sequence of numbers that appear to be random and at the same time has interesting statistical properties and the right distribution. Since a random number generator does a calculation with finite information, it will eventually repeat itself. This brings in the requirement that the period of repetition should be sufficiently long.

A random number generator stores an internal state with which it calculates the next random number and the next internal state. The state should be large enough to allow a long period. The random number is initialized with a number called its seed. Initializing it again with the same seed should give the same sequence of random numbers. If we do not want the same sequence, we can initialize it with information that will never be the same, such as the current date and time.

It turns out that a good simple method is the linear congruential generator. If x is the internal state and s is the seed, then the internal state is updated as follows:

x0 = s
xn = (axn-1 + b) mod m

The constants a, b and m have to be carefully chosen so that the generated sequence has the good properties that we discussed earlier. It is easy to implement this generator:

slogan> define make_rand = let (a = 333667, b = 213453321) 
                            fn (seed, max) 
                             fn () { seed = modulo(a * seed + b, max); seed };

Make_rand returns a random number generator function that is initialized with seed. The function makes sure that the numbers returned are within the max limit:

slogan> define r = make_rand(15, 10); 
slogan> r(); 
=> 6
slogan> r(); 
=> 3
A functional random number generator

It is possible to implement the random number generator using lazy streams. Instead of using a variable, the state is stored in a recursive argument of the function calls:

slogan> define make_rand = let (a = 333667, b = 213453321) 
                            fn (max) 
                             letrec (r = fn (seed) 
                                          let (x = modulo(a * seed + b, max)) 
                                           x : delay(r(x))) 
                              r;

One difference with the stateful implementation is that the seed argument has moved from the constructor to the random number generator:

slogan> define r = make_rand(10); 
slogan> define s = r(15); 
slogan> s; 
=> [6 <promise>]
slogan> force(tail(s)); 
=> [3 <promise>]

Limitations of stateful programming

Two major limitations of programming with explicit state are:

  1. Stateful programs are sequential. In contrast, the real world is stateful and parallel. In the next chapter we will see how explicit state and concurrency can go together in the same program.
  2. The real world is distributed and explicit state is hard to use well in a distributed system.
