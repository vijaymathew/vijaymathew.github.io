Lazy Evaluation

Until now we have evaluated expressions in order, from left to right. This is known as eager evaluation and is the default evaluation strategy in almost all programming languages. A second evaluation strategy called lazy evaluation is also possible. In lazy evaluation an expression is evaluated only when its value is needed. Consider the following program:

slogan> function f1(x) delay({ showln("f1"); 1 + x * (3 + x * (3 + x)) }); 
slogan> function f2(x) delay({ showln("f2"); let (y = x * x) y * y }); 
slogan> function f3(x) delay({ showln("f3"); x * 1 + x * 1 }); 

slogan> define a = f1(10); 
slogan> define b = f2(20); 
slogan> define c = f3(30); 
slogan> define d = force(a) + force(b); 
-> f1
   f2
slogan> d;
=> 161331

The special function delay takes an expression as its argument and returns an object representing a promise to execute the expression sometime in the future. When the program needs the value of the expression, the promise object is forced to evaluate it. For this we call the force function. As you can see from the previous example, even though a call is made to f3, the expression delayed by it is not executed.

Lazy evaluation is a powerful concept that can simplify many programming tasks. For instance, it can help in the design of functional algorithms that have good amortized or worst-case time bounds. With lazy evaluation we can write producer/consumer pairs where the consumer decides how much data needs to be read from the stream. Lazy evaluation can also interact in interesting ways with concurrency. In this chapter we introduce all these ideas and more.
