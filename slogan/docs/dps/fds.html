Functional Data Structures

This chapter introduces the design and implementation of functional data structures through examples. These data structures do not depend in crucial ways on assignments, which are discouraged in functional programming.

Queues

Let us start by studying the implementation of a simple data structure - queue. A queue is a sequence of elements with an enqueue and a dequeue operation. The enqueue operation adds an element to one end of the queue and the dequeue operation removes an element from the other end. An additional operation front can also be defined to inspect the element at the end. We say the queue has FIFO (first-in, first-out) behavior. Let us investigate how to program queues in the functional world.

A naive queue implementation using lists is shown below:

slogan> function make_queue() []; 
slogan> function enqueue(q, x) [x q]; 
slogan> function dequeue(q) take(length(q) - 1, q); 
slogan> function front(q) at(length(q) - 1, q); 

slogan> define q = enqueue(enqueue(enqueue(make_queue(), 30), 20), 10); 
slogan> q;
=> [10, 20, 30]
slogan> front(q); 
=> 30
slogan> dequeue(q); 
=> [10, 20]

The problem with this implementation is that both dequeue and front are slow. They take time proportional to the length of the list. Is it possible to define dequeue and front to be of constant-time, like enqueue?. In strict functional style, i.e., without assignments, we cannot achieve it. The best we can do is to get amortized constant-time. That is, we can perform any sequence of n enqueue and dequeue operations in some constant time relative to n. Any individual operation might not be constant-time, however.

Here is the definition of a queue with amortized constant-time enqueue and dequeue:

slogan> function make_queue() [[] []]; 

slogan> function check(q) 
          match (q) 
           [[] r]: [reverse(r) []], 
           else: q; 

slogan> function enqueue(q, x) 
          match (q) 
           [f r]: check([f [x r]]);

slogan> function dequeue(q) 
          match (q) 
           [f r]: check([tail (f) r]);

slogan> function front(q) 
          match (q) 
           [f r]: head(f);

This implementation uses the pair (F R) to represent the queue. F and R are lists. F represents the front of the queue and R represents the back of the queue in reversed form. An element is inserted by adding it to the front of R and deleted by removing it from the front of F. For example, if F=[a b] and R=[d c], deleting the first element returns a and makes F=[b]. Inserting the element e makes R=[e d c]. Both operations are constant-time.

We move the elements in R to F when F becomes empty. This is is done by the check function which does a list reverse operation on R. The reverse takes time proportional to the length of R. Each element that goes through the queue is passed exactly once from R to F. Allocating the execution time of reverse to each element therefore gives a constant time per element, making the queue amortized.

Trees

Next to linear data structures such as lists and queues, trees are the most important recursive data structure in a programmer's repertory. A tree is either a leaf node or a node that contains one or more trees.

There exist many varieties of trees, the main difference being in the branching structures. A unary tree is a tree in which non-leaf nodes always have exactly one sub-tree. A list is also a unary tree. A binary tree always have exactly two sub-trees, a ternary tree have exactly three sub-trees an so on.

In this section we give the basic algorithms for constructing and traversing trees. We also look at how to insert, lookup and delete information in trees. We define these algorithms in the context of a type of binary tree called ordered binary tree.

We can use lists to represent tree nodes but in this section we will use records. A record is a way to group together related values, similar to structs in the C language.

In an ordered binary tree each non-leaf node includes a key-value pair. Boolean comparison functions defined on the key is used to determine its position in the tree. An equality function defined on the key is used to retrieve the value or information associated with the key. A non-leaf node in an ordered binary tree needs to have four components: a key and its associated value, the left sub-tree and the right sub-tree. This structure is defined as:

slogan> record ob_tree(key, value, left, right);

(When we define a record, a constructor function with the same name as the record is created for us. Functions for accessing and updating the member values are also automatically defined.)

As they carry no additional information, leaf-nodes can be represented by the symbol !leaf.

We will also define a function that makes defining non-leaf nodes easier. As a non-leaf node in itself is a tree, we will call this function tree:

slogan> function tree(k, v, left = !leaf, right = !leaf) 
          ob_tree(key = k, value = v, left = left, right = right);

Now we are ready to create and use ordered binary trees:

slogan> define simple_tree = tree(0, 100); 

slogan> ob_tree_key(simple_tree); 
=> 0
slogan> ob_tree_value(simple_tree); 
=> 100
Storing and finding information

To insert information in an ordered binary tree, we construct a new tree that is identical to the original except that it has one more node with the new key-value pair. The insertion operation is defined as:

slogan> function insert(k, v, t) 
          match (t) 
           !leaf: tree(k, v),
           ob_tree(key, value, left, right) where k == key: t,
           @ where k < key: tree(key, value, insert(k, v, left), right), 
           @ where k > key: tree(key, value, left, insert(k, v, right));

(In a match expression, the @ symbol is used to repeat the previous pattern.)

Insert is defined according to the orderedness condition of the keys. If the key of the node to be inserted is less than the key of the parent node, it is inserted on the left, otherwise it is inserted on the right. This enables the search algorithm to eliminate half the remaining nodes at each step. This is called binary search. We implement binary search on a tree as follows:

slogan> function lookup(k, t) 
          match (t) 
           !leaf: !notfound,
           ob_tree(key, value, left, right): 
            if (k < key) lookup(k, left) 
            else if (k > key) lookup(k, right) 
            else [!found value];

Let us try out the insert and lookup functions:

slogan> define fifa_rankings = insert(5, !Belgium, 
                                insert(2, !Argentina, 
                                 insert(4, !Colombia, 
                                  insert(3, !Netherlands, 
                                   tree(1, !Germany))))); 

slogan> lookup(2, fifa_rankings);
=> [found Argentina]

slogan> lookup(4, fifa_rankings);
=> [found Colombia]

slogan> lookup(7, fifa_rankings);
=> notfound

(Note: FIFA rankings used may not be upto date, but that is not the point!)

Deleting information

Deletion is not as simple as insertion or lookup. We have to make sure that deleting a node will not lead to the removal of the entire sub-tree below it. We implement the delete function following the same pattern as lookup. Once we have found the key to delete, we replace it with the smallest key from one of its sub-trees. We will define a helper function, remove_smallest that will return the smallest key from a tree and a new tree that lacks this key:

slogan> function remove_smallest(t) 
          match (t) 
           !leaf: [], 
           ob_tree(key, value, left, right): 
             { match (remove_smallest(left)) 
                 []: [[key value] right], 
                 [[kp vp] tp]: [[kp vp] tree(key, value, tp, right)] }, 
           else: t;

Now we can define delete in terms of remove_smallest:

slogan> function delete(k, t) 
          match (t) 
           !leaf: !leaf, 
           ob_tree(key, value, left, right) where k == key: 
             { match (remove_smallest(right)) 
                []: left, 
                [[kp vp] tp]: tree(kp, vp, left, tp) }, 
           @ where k < key: tree(key, value, delete(k, left), right), 
           @ where k > key: tree(key, value, left, delete(k, right));
Traversing trees

Traversing a tree means to perform an operation on its nodes in some well-defined order. There are many ways to traverse a tree. Many of these are derived from one of two basic traversals, called depth-first and breadth-first traversal. Let us look at these traversals.

Depth-first is the simplest traversal. For each node, it visits first the node itself, then the leftmost sub-tree, and then the rightmost sub-tree. The following function implements depth-first traversal. It calls a function f on each node it visits and accumulates the results into a list:

 slogan> function dfs(t, f) 
           letrec (dfs_loop = fn (t , f, accum) 
                               match (t) 
                                !leaf: reverse(accum), 
                                ob_tree(k, v, l, r): 
                                 dfs_loop(r, f, dfs_loop(l, f, [f(k, v) accum])))
           dfs_loop(t, f, []);

The following call to dfs collects all keys and values in a tree into a list:

slogan> dfs(fifa_rankings, fn(k, v) [k v]); 
=> [[5 Belgium], [1 Germany], [3 Netherlands], [2 Argentina], [4 Colombia]]

Breadth-first is a second basic traversal. It first traverses all nodes at depth 0, then all nodes at depth 1, and so forth, going one level deeper at a time. At each level, it traverses the nodes from left to right. The depth of a node is the length of the path from the root to the current node, not including the current node. To implement breadth-first traversal, we need a queue to keep track of all the nodes at a given depth. The implementation given below, called bfs, uses the queue data type we defined in the beginning of this chapter. The next node to visit comes from the head of the queue. The node's two sub-trees are added to the tail of the queue. The traversal will get around to visiting them when all the other nodes of the queue have been visited, i.e., all the nodes at the current depth. bfs also takes a function that will be called for each node encountered and it returns all the resulting values accumulated into a list.

slogan> function bfs(t, f) 
          letrec (bfs_loop = fn (f, q, accum) 
           if (is_empty(q)) reverse(accum) 
           else match (front(q)) 
                 !leaf: bfs_loop(f, dequeue(q), accum), 
                 ob_tree(k, v, l, r): 
                  bfs_loop(f, enqueue(enqueue(dequeue(q), r), l), [f(k, v) accum])) 
          bfs_loop(f, enqueue(make_queue(), t), []);

We have defined all the important operations for the ordered binary tree. Our tree still have one limitation - it can have only numbers as keys. As an exercise, modify the tree implementation in such a way that it can have keys of any type (strings, symbols, records, and so on).
