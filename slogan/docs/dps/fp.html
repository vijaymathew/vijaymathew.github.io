Basic Functional Programming Techniques

We say an operation is functional if, whenever called with the same input, it returns the same results independent of any other computation state. A functional operation is:

  • independent - does not depend on any execution state outside of itself.
  • stateless - has no internal execution state that is remembered between calls.
  • deterministic - always gives the same results when given the same arguments.

Functional programming is important because of two properties:

  • Functional programs are compositional. A functional program consists of components that can each be written, tested, and proved correct independently of other components and of its own past history (previous calls).
  • Reasoning about functional programs is simple. Programs written in the functional style are easier to reason about than programs written in other styles. Since functional programs compute only values, simple algebraic and logical reasoning techniques can be used to prove their correctness.

These two properties are important both for programming in the large and in the small. It would be nice if all programs could easily be written in the functional style. Unfortunately, this is not the case. The functional style is a good fit for certain kinds of programs and a bad fit for others. This chapter examine the most important functional programming techniques and explain what kinds of programs can and cannot be easily written in this style.

The basic technique for writing functional programs is to consider the program as a set of recursive function definitions, using higher-order programming to simplify the program structure. A recursive function is one whose definition body refers to the function itself, either directly or indirectly. Direct recursion means that the function itself is used in the body. Indirect recursion means that the function refers to another function that directly or indirectly refers to the original function. Higher-order programming means that functions can have other functions as arguments and results. This ability underlies all the techniques for building abstractions that we will encounter in this tutorial.

Iterative computation

We will start our study of functional programming by looking at a very simple kind of program - the iterative computation. Most iterative computations follow this pattern:

  1. Start with an initial state S0.
  2. Transform the state in successive steps until reaching the final state - Sfinal.

This pattern can be expressed as a function:

slogan> function iterate(state, is_done, transform) 
          if (is_done(state)) state 
          else iterate(transform(state), is_done, transform);

The arguments is_done and transform are two problem-dependent functions. iterate first checks if the final state has reached by calling is_done. If is_done returns true, the current state is returned. Otherwise, iterate calls itself with the next state. As this is the last thing that iterate does it is treated as a special case of recursion known as a tail call. In such situations, the program does not need to return to the calling function when the called function ends. Therefore, after the tail call, there is no need for the called function to keep any information about the calling function in the stack. Slogan take advantage of this fact and actually do not use any extra stack space when doing a tail call. We say that Slogan support proper tail calls. Because of tail calls, there is no limit to the number of times iterate can call itself. This feature makes special looping constructs like for and while appear superfluous.

Let us see how iterate can stand-in for a traditional for loop:

slogan> iterate(0, fn(i) i >= 3, fn(i) {showln(i); add1(i)}); 
-> 0
   1
   2
   3

A good example of iterative computation is Newton's method for calculating the square root of a positive real number x. The idea is to start with a guess of the square root, and to improve this guess iteratively until it is accurate enough. Here is one way of defining Newton's method as an iterative computation:

slogan> function sqrt(x)
          let (guess = 1.0) 
           sqrt_iter(guess, x); 

slogan> function sqrt_iter(guess, x) 
          if (is_good_enough(guess, x)) guess 
          else sqrt_iter(improve(guess, x), x); 

slogan> function improve(guess, x)
          (guess + x / guess) / 2.0; 

slogan> function is_good_enough(guess, x) 
          abs(x - guess * guess) / x < 0.00001;


slogan> sqrt(10);
=> 3.162277665175675
Local functions

The sqrt function uses several "helper" functions: sqrt_iter, improve and is_good_enough. Generally, it is not a good practice to define such helper functions in the global scope. The basic principle is that a helper function defined only as an aid to define another function should not be visible elsewhere.

In the square root example, sqrt_iter is only needed inside sqrt, and improve and good_enough are only needed inside sqrt_iter. There are two basic ways to express this visibility, with somewhat different semantics. The first way is to define the helper functions outside of sqrt in a letrec expression. (letrec is an extension of let that allows a definition to refer recursively to itself and other local definitions.):

slogan> define sqrt = 
          letrec (improve = fn(guess, x)  
                             (guess + x / guess) / 2.0,
 
                  is_good_enough = fn(guess, x) 
                                    abs(x - guess * guess) / x < 0.00001, 

                  sqrt_iter = fn(guess, x)  
                               if (is_good_enough(guess, x)) guess 
                               else sqrt_iter(improve(guess, x), x)) 

            function(x)
              let (guess = 1.0) 
                sqrt_iter(guess, x);

improve and is_good_enough could be defined local to sqrt_iter only. This would result in two levels of let expressions, which is harder to read. By putting all three helper functions in the same top-level let expression, we have traded visibility for readability. Putting helper functions inside the function that uses then lets them see the arguments of the main function. Therefore, they need fewer arguments. But each time the main function is invoked, new helper functions are created. All these considerations leads to the following definition of sqrt which tries to balance efficiency with visibility:

slogan> function sqrt(x) 
          letrec (improve = fn(guess) 
                             (guess + x / guess) / 2.0, 

                  is_good_enough = fn(guess) 
                                    abs(x - guess * guess) / x < 0.00001, 

                  sqrt_iter = fn(guess) 
                               if (is_good_enough(guess) ) guess 
                               else sqrt_iter(improve(guess)), 

                  guess = 1.0) 

            sqrt_iter(guess);

We started this section by defining a generic iterate function. We can use it instead of sqrt_iter by passing it the functions is_good_enough and improve to define a newer and shorter version of sqrt:

slogan> function sqrt(x) 
          iterate(1.0, 
                  fn(guess) abs(x - guess * guess) / x < 0.00001, 
                  fn(guess) (guess + x / guess) / 2.0);

Recursive computation

Iterative computations are a special case of a more general kind of computation called recursive computation. Recursive computations are more general because a recursive function can call itself anywhere in the body and can call itself more than once, while an iterative call must be the last thing the function does. Unlike iterative function calls, a recursive function call can grow stack size because it has to keep track of a place to return in the current function after the recursive call finishes. Consider the definition of the factorial function from the Introduction:

slogan> function factorial(n) 
          if (n == 0) 1 
          else n * factorial(n - 1);

This definition gives a computation whose maximum stack size is proportional to the argument n. In the else part of the function, it has to build up a chain of deferred multiplications whose length is determined by n. For very large numbers, this chain can outgrow the available memory and the function will fail. It is possible to make factorial more efficient by introducing an additional argument that keeps track of the result computed so far. This extra argument, known as an accumulator, will enable us to rewrite factorial as an iterative computation that will not run out of stack space:

slogan> function factorial(n) 
          letrec (fact_iter = fn (n , a) 
                               if (n == 0) a
                               else fact_iter(n - 1, a * n)) 
           fact_iter(n, 1);
Recursive programming with lists

In programming, recursion occurs in two major ways: in functions and in data types. A function is recursive if its definition has at least one call to itself. A data type is recursive if it is defined in terms of itself. For example, a list is defined in terms of a smaller list. The two forms of recursion are strongly related since recursive functions can be used to calculate with recursive data types.

List values are very concise to create and to take apart, yet they are powerful enough to encode any kind of complex data structure. Because of lists' simple structure, functional programming with them is easy and powerful. This section gives the basic techniques of programming with lists.

A list is a recursive data structure, i.e., it is defined in terms of a smaller version of itself. To write a function that calculates on lists we have to follow this recursive structure. The function consists of two parts:

  • A base case. For small lists (say, of zero, one, or two elements), the function computes the answer directly.
  • A recursive case. For bigger lists, the function computes the result in terms of the results of one or more smaller lists.

Slogan has a built-in function length that returns the number of elements in a list. As a simple example, let us redefine this function following the above technique:

slogan> function length(ls) 
          match (ls) 
           []: 0, 
           [_ xs]: 1 + length(xs); 

slogan> length([10, 90, 0, 100]);
=> 4
slogan> length(['a', 'b', 'c']); 
=> 3
slogan> length([10, 90, ['a', 'b', 'c'], 0, 100]); 
=> 5

Can you define an iterative version of length using an accumulator argument?

Now let us define an "advanced" version of length that counts the number of elements in nested lists:

slogan> function length_n(ls) 
          match (ls) 
           []: 0, 
           [x xs] where is_list(x): length_n(x) + length_n(xs), 
           [_ xs]: 1 + length_n(xs); 

slogan> length_n([10, 90, ['a', 'b', 'c'], 0, 100]); 
=> 7

The second clause of match contains a guard which is introduced by the keyword where. The clause is executed only if the pattern match succeeds and the expression in the guard evaluates to true.

Both the length functions that we defined displays a curious property: The recursive structure of the functions exactly follows the recursive structure of the list data type. This leads us to an important principle in designing functional programs: A function that calculates with a given type almost always has a recursive structure that closely mirrors the type definition. But naively following the recursive structure of a data type to define functions can lead to inefficient implementations. This can be made clear with an example. Let us define a function to reverse the elements of a list:

slogan> function reverse(ls) 
          match (ls) 
           []: [], 
           [x xs]: append(reverse(xs), [x]);

Is this function efficient? Let us do an intuitive analysis of the function. For a list of length n there will be n recursive calls followed by n calls to append. Each append call will have a list of length n/2 on average. The total execution time is therefore proportional to n * n/2, namely n2. This is rather slow. We would expect that reversing a list, which is not exactly a complex calculation, would take time proportional to the input length and not to its square.

This function has a second defect: the stack size grows with the input list length, i.e., it defines a recursive computation that is not iterative. Naively following the recursive definition of reverse has given us a rather inefficient result! Luckily, we can use the accumulator technique that we learned earlier for getting around both these inefficiencies. Here the accumulator contains the reverse of the part of the list already seen. Updating the accumulator is easy: just put a new list element in front. The initial state is the empty list []. This gives the following version of reverse:

slogan> function reverse(ls) 
          letrec (iter_reverse = fn (ls, result) 
                                  match (ls) 
                                   []: result, 
                                   [x xs]: iter_reverse(xs, [x result])) 
           iter_reverse(ls, []);

This version of reverse is both a linear-time and an iterative computation.

The reverse function comes built-in with Slogan, along with a suite of functions useful for working with lists. These functions are documented in the Programmer's Manual.

Higher-order programming

Higher-order programming is the collection of programming techniques that become available when a language support functions as first-class values. The term higher order comes from the concept of order of a function. A function all of whose arguments are not functions is of first order. A language that allows only this kind of functions is called a first-order language. A function that has at least one first-order function in an argument is of second order, and so forth. A language that allows programming with higher-order functions is called a higher-order language.

The four basic operations that underlie all the techniques of higher-order programming are:

  1. Procedural abstraction: the ability to convert any expression into a function value.
  2. Generecity: the ability to pass function values as arguments to a function call.
  3. Instantiation: the ability to return function values as results from a function call.
  4. Embedding: the ability to put function values in data structures.

Procedural abstraction

We have already introduced procedural abstraction. Any expression expr can be "packaged" into a function by writing it as function() expr. This does not execute expr but creates a function value that can be used to execute expr at some time in the future. Function values allow more than just delaying execution of a statement. They can have arguments, which allows some of their behavior to be influenced by the call. Along with the expression, the function also packages its "context", making sure that executing expr always gives the same result. An expression packaged along with its contextual environment is known as a closure. This is a principle tool for building abstractions and underlies both higher-order programming and object-oriented programming. Following example shows a simple closure in action:

slogan> define f = let (a = 10) 
                    function () a + 100;

The expression a + 10 is packaged into a function which is bound to f. No matter where we call it, f executes the expression in a context where a is bound to 10:

slogan> f();
=> 110
slogan> let (a = 200) f();
=> 110
Genericity

To make a function generic is to let any specific entity (i.e., any operation or value) in the function body become an argument of the function. We say the entity is abstracted out of the function body. The specific entity is given when the function is called. Each time the function is called another entity can be given. We saw an example of a generic function when we defined iterate earlier in this chapter. It takes two function arguments transform and is_done that influences its behavior each time it is called.

Let us look at a second example of a generic function. Consider the function sum_list:

slogan> function sum_list(ls) 
          match (ls) 
           []: 0, 
           [x xs]: x + sum_list(xs);

This function has two specific entities: the number zero (0) and the operation plus (+). The zero is a neutral element for the plus operation. If we abstract out these two entities and give them as arguments, we get a generic function that can be used to "reduce" a list to a single value. Slogan has such a function called fold_right - a function that "folds" a list from right to left to produce a single value. (fold_left is the left-associative version of fold_right). If we use fold_right, we get a concise definition of sum_list:

slogan> function sum_list(ls) 
          fold_right(add, 0, ls);

We can use fold_right to define other functions on lists. Here is a function that returns true if there is at least one true in the list:

slogan> function has_any(ls) 
          fold_right(fn(x, y) x || y, false, ls); 

slogan> has_any([false, true, false]);
=> true
slogan> has_any([1 == 2, 1 < 2]);
=> true
slogan> has_any([1 == 2, 1 > 2]);
=> false
Instantiation

An example of instantiation is the "counter" factory that we met in the Introduction. Another example is a function that returns a sorting function. The function make_sort takes a boolean comparison function cf and returns a function that can be used to sort a list which uses cf to order the elements:

slogan> function make_sort(cf) 
          function (ls) sort(ls, test = cf); 

slogan> define num_sorter = make_sort(is_number_lt);
slogan> define char_sorter = make_sort(is_char_lt); 
slogan> num_sorter([10, 8, 0, 34]);
=> [0, 8, 10, 34]
slogan> char_sorter(['r','w', 'a', 'd']);
=> ['a', 'd', 'r', 'w']

(Is_number_lt returns true if the first numeric argument is less than the second numeric argument. is_char_lt is an equivalent function for comparing characters.)

Embedding

Putting function values in data structures has many uses. For example, this can be used to build lazy data structures: data structures that are built incrementally on demand. The idea is to build only a small part of the data structure with functions at the extremities that can be called to build more. We will use embedded functions in creating many abstractions later in this tutorial.

Traversing lists

Higher-order programming is often used together with lists. A common task is to traverse a list and calculate a new list based on this traversal. A higher-order function that can be used for this task is map, which produces a new list by applying a function to each element in the old list:

slogan> map(sqrt, [10, 3, 1.5]); 
=> [3.1622776601683795, 1.7320508075688772, 1.224744871391589]

Here is another example that uses map to perform simple string encryption and decryption:

slogan> function encrypt_char(c) integer_to_char(char_to_integer(c) + 3);
slogan> function decrypt_char(c) integer_to_char(char_to_integer(c) - 3);
slogan> function encrypt_string(s) list_to_string(map(encrypt_char, string_to_list(s))); 
slogan> function decrypt_string(s) list_to_string(map(decrypt_char, string_to_list(s))); 

slogan> encrypt_string("hello, world"); 
=> "khoor/#zruog"
slogan> decrypt_string(encrypt_string("hello, world")); 
=> "hello, world"

Sometimes we want to apply a function on each element in the list, without producing a result list. The iterative function for_each enables us to do this. Say, we want to print the square roots of all numbers in a list: 

slogan> for_each(fn(x) showln(sqrt(x)), [10, 3,1.5]); 
-> 3.1622776601683795
   1.7320508075688772
   1.224744871391589

Another common list traversing operation is filter, which applies a boolean function to each list element and outputs the list of all elements that give true. The following program filters out all odd numbers from a list:

slogan> filter(is_odd, [1, 2, 3, 4, 5]); 
=> [1, 3, 5]
