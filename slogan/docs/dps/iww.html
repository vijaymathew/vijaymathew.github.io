Interacting with the World

Pure functional programming is somewhat detached from the real world where entities have changing state. To connect a functional program to the real world, some non-functional operations are needed. This chapter covers few such operations - input/output on files and the network, invoking code written in C and compiling Slogan programs into applications run standalone on the host operating system. We also introduce the module system which helps in organizing large programs.

File and Network IO

A simple way to interface functional programming with the real world is by using files. A file is a sequence of values that is stored external to the program on a permanent storage medium such as a hard disk. A text file is a file containing a sequence of characters. Text files are structured as a sequence of lines, each line ending with the newline character. A binary file contains a sequence of bytes that may represent any type of data. The contents of a binary file has to be interpreted by the program reading it. Both text and binary files mark their end with a special object known as the eof (end-of-file) object.

Reading and writing files

Slogan does all its input and output operations through objects called ports. We read and write files by opening a port on it. A port may be an input port, an output port, or both simultaneously.

A file is read by first opening an input port on it. This is achieved with the open_input_file function. On the input file port, we can either call the read_char or read_byte function to fetch a character or a byte respectively. After using the port, it has to be closed by calling close_port. To read complete text file, we can use the read_all_lines function. To read a complete binary file, call the read_all_bytes function. It returns a list of all bytes in the file. Now as we have met all the functions required for reading a file, let us actually read in a text file and view its contents:

// Assuming that you have a file called "hello.txt" with a couple of lines of text in it:
slogan> define in_file = open_input_file( "hello.txt"); 
slogan> read_all_lines(in_file); 
// => You will see the contents of "hello.txt" here.
slogan> close_port(in_file);

Writing a file is equally easy. You open a file output port with open_output_file, write characters or bytes using the write_char or write_byte functions and then close the port. You can also call the show or showln functions for writing file by passing the port as a keyword argument:

slogan> define out_file = open_output_file("out.txt"); 
slogan> showln(port = out_file, "hello from Slogan"); 
slogan> showln(port = out_file, "2 * 3 = ", 2 * 3);            
slogan> close_port(out_file);

The call to close_port is important because it makes sure that the operating system resources associated with the port are released after use. Forgetting to close the port will lead to a resource leak. There is a convenience function called call_with_port that can save us from this trouble. It takes a port and a function as arguments. It calls the function by passing it the port. Once the function returns, call_with_port closes the port. It ensures that the port is closed even if the function returns abnormally by raising an exception. Let us see call_with_port by writing a function that makes a copy of a text file:

slogan> function copy(src, dest) 
          call_with_port(open_input_file(src), 
            fn(in) call_with_port(open_output_file(dest), 
                     fn(out) show(port = out, read_all_lines(in))));

(Slogan has a built-in function - copy_file - that you can use to copy any type of file.)

TCP ports

We can read files on remote servers using TCP ports. The following example shows how to read a file hosted on a HTTP server:

slogan> call_with_port(open_tcp_client_port("www.google.com" , port_number = 80), 
                       fn(p) { showln(port = p, "GET /search?q=tcp"); 
                               flush_output_port(p); 
                               read_all_lines(p) });

The function flush_output_port makes sure that the data is flushed out from the TCP client ports buffer and pushed to the server. When you run this code, you should see the first results page returned by Google for the query "tcp".

We can use TCP ports to write network servers as well. Here is a simple echo server that sends back whatever the client sends it:

slogan> call_with_port(open_tcp_server_port(9999), 
                       fn(p) let (client = read(p)) { showln(port = client, read_line(client)); 
                                                      close_port(client) });

The read function call accepts a new client connection on the server port.
