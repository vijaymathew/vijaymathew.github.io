Interacting with the World

Pure functional programming is somewhat detached from the real world where entities have changing state. To connect a functional program to the real world, some non-functional operations are needed. This chapter covers few such operations - input/output on files and the network, invoking code written in C and compiling Slogan programs into applications run standalone on the host operating system. We also introduce the module system which helps in organizing large programs.

File and Network IO

A simple way to interface functional programming with the real world is by using files. A file is a sequence of values that is stored external to the program on a permanent storage medium such as a hard disk. A text file is a file containing a sequence of characters. Text files are structured as a sequence of lines, each line ending with the newline character. A binary file contains a sequence of bytes that may represent any type of data. The contents of a binary file has to be interpreted by the program reading it. Both text and binary files mark their end with a special object known as the eof (end-of-file) object.

Reading and writing files

Slogan does all its input and output operations through objects called ports. We read and write files by opening a port on it. A port may be an input port, an output port, or both simultaneously.

A file is read by first opening an input port on it. This is achieved with the open_input_file function. On the input file port, we can either call the read_char or read_byte function to fetch a character or a byte respectively. After using the port, it has to be closed by calling close_port. To read complete text file, we can use the read_all_lines function. To read a complete binary file, call the read_all_bytes function. It returns a list of all bytes in the file. Now as we have met all the functions required for reading a file, let us actually read in a text file and view its contents:

// Assuming that you have a file called "hello.txt" with a couple of lines of text in it:
slogan> define in_file = open_input_file( "hello.txt"); 
slogan> read_all_lines(in_file); 
// => You will see the contents of "hello.txt" here.
slogan> close_port(in_file);

Writing a file is equally easy. You open a file output port with open_output_file, write characters or bytes using the write_char or write_byte functions and then close the port. You can also call the show or showln functions for writing file by passing the port as a keyword argument:

slogan> define out_file = open_output_file("out.txt"); 
slogan> showln(port = out_file, "hello from Slogan"); 
slogan> showln(port = out_file, "2 * 3 = ", 2 * 3);            
slogan> close_port(out_file);

The call to close_port is important because it makes sure that the operating system resources associated with the port are released after use. Forgetting to close the port will lead to a resource leak. There is a convenience function called call_with_port that can save us from this trouble. It takes a port and a function as arguments. It calls the function by passing it the port. Once the function returns, call_with_port closes the port. It ensures that the port is closed even if the function returns abnormally by raising an exception. Let us see call_with_port by writing a function that makes a copy of a text file:

slogan> function copy(src, dest) 
          call_with_port(open_input_file(src), 
            fn(in) call_with_port(open_output_file(dest), 
                     fn(out) show(port = out, read_all_lines(in))));

(Slogan has a built-in function - copy_file - that you can use to copy any type of file.)

TCP ports

We can read files on remote servers using TCP ports. The following example shows how to read a file hosted on a HTTP server:

slogan> call_with_port(open_tcp_client_port("www.google.com" , port_number = 80), 
                       fn(p) { showln(port = p, "GET /search?q=tcp"); 
                               flush_output_port(p); 
                               read_all_lines(p) });

The function flush_output_port makes sure that the data is flushed out from the TCP client ports buffer and pushed to the server. When you run this code, you should see the first results page returned by Google for the query "tcp".

We can use TCP ports to write network servers as well. Here is a simple echo server that sends back whatever the client sends it:

slogan> call_with_port(open_tcp_server_port(9999), 
                       fn(p) let (client = read(p)) { showln(port = client, read_line(client)); 
                                                      close_port(client) });

The read function call accepts a new client connection on the server port.

Now you might want to test our echo server by writing a client for it.

FFI

FFI or the Foreign Function Interface is a set of functions that allows our Slogan programs to call functions in a dynamically loadable C library. (Known as shared libraries in the Unix world). To be callable from Slogan, the C functions must follow certain conventions. Its arguments must be of type ___slogan_obj. It has to use type conversion functions defined in slogan.h to convert a ___slogan_obj to the appropriate C type. These C types are currently supported: char, bool, short, int, int32, int64, float, double and void pointer. The size of all types except int32 and int64 are system dependent. The C function can directly return all of these types in addition to ___slogan_obj.

The following C program defines two functions that can be called from Slogan. The first function - add - expects two integers as arguments and returns their sum. The second function - say_hello - expects a string as argument and returns a C string:

// demo_lib.c

#include <stdio.h>
#include "slogan.h"

int add(___slogan_obj a, ___slogan_obj b)
{
  int ia, ib;
  ___slogan_obj_to_int(a, &ia);
  ___slogan_obj_to_int(b, &ib);
  return ia + ib;
}

char *say_hello(___slogan_obj msg)
{
  char *s;

  ___slogan_obj_to_charstring(msg, &s);
  printf("Slogan says: %s\n", s);
  return "hi from C";
}

This program can be compiled into a shared library using the GNU C Compiler using this command: 

gcc -Wall -shared -I$SLOGAN_ROOT/src/include -I$SLOGAN_ROOT/platform/gsc/include -fPIC -o demo_lib.so demo_lib.c

To load the library into slogan, call the ffi_open function with the library file name. This will give you a handle object that you can use to call functions in the library. A group of functions named in the format - ffi_call_RT_N - is used to invoke C functions. RT denotes the return type of the C function and N specifies the number of arguments the function takes. You can release the library handle by calling ffi_close. The following program shows how to load and use our demo_lib shared library:

slogan> define clib = ffi_open("./demo_lib.so"); 
slogan> ffi_call_int_2(clib, "add", 10, 20); 
=> 30
slogan> ffi_call_char_string_1(clib, "say_hello", "hi from Slogan"); 
-> Slogan says: hi from Slogan
=> hi from C
slogan> ffi_close(clib);

Modules and Compilation

As a program grows in size, we will need some way to orgnaize it. Writing the program as one big monolithic whole can be confusing. A better way is to partition the program into logical units, each of which implements a set of operations that are related in some way. Each logical unit has two parts, an interface and an implementation. Only the interface is visible from outside the logical unit. A logical unit may use others as part of its implementation.

Slogan provides a module mechanism to group together related operations into an entity that has an interface and an implementation. The following example shows a dictionary module that can be used to associate a word with some value. The dictionary module exports three operations to the outside world:

  1. create - creates a new empty dictionary.
  2. put - adds a word and its associated value to the dictionary.
  3. get - retrieves the value associated with a word.

Exactly how these operations are implemented is hidden from the world. Someone who uses the dictionary module onle need to know about the functionality provided by the three exported operations. The following implementation of dictionary uses the built-in hashtable data structure to associate words with values. This time let us write and save the code in a source file called "dict.sn":

// file: dict.sn
module dictionary export (create) { 
  define table = make_hashtable(string_hash, is_string_eq); 
  module create export  (put, get) { 
    function put(w, v) hashtable_set(table, w, v); 
    function get(w) hashtable_at(table, w) 
  }; 
}; 

define dict = dictionary();

The dictionary module is defined in two layers. The top layers exports the symbol create which refers to an internal module where the put and get functions are implemented. As you might have already figured out, a module is nothing but a closure. Each time dictionary is called, it returns a closure with a new hashtable and a new instance of the create module that refers to that hash table.

To use the dictionary module, we have to first load the file "dict.sn" into the REPL:

slogan> load "dict"; 

slogan> define d = dict.create(); 
slogan> d.put("name", "sam"); 
slogan> d.put("age", 2);          
slogan> showln(d.get("name"), " is ", d.get("age"), " years old."); 
-> sam is 2 years old.

We will write a new module that uses the features exported by dictionary. This is a module that can count the words read from a text input port:

// file: wc.sn 
load "dict"; 

module word_count(port) export (count) locals (words_dict) { 
  function count(word) { 
    if (not(words_dict)) 
     words_dict= init_words_dict(string_split(read_all_lines(port)), dict.create()); 
    words_dict.get(word) 
  }; 

  function init_words_dict(words, wd) 
    if (is_empty(words)) wd 
    else let (count = wd.get(head(words))) { 
           if (not(count) ) count = 0; 
           wd.put(head(words), count + 1); 
           init_words_dict(tail(words), wd)
         }; 
};

The word count program presents a new feature - locals - which are similar to module arguments. Both module arguments and locals are shared by the functions defined in the module. The difference is that, while the users of the module can pass in values for the arguments, locals have to be initialized in the module itself.

Let us complete this section by writing a script that uses the word_count module to find the number of occurences in a file. The file to read and the word to count is passed as command line arguments to the script:

// file: fwc.sn 
load "wc" ; 

define args = command_line(); 
define p = open_file_input_port(at(args, 1)); 
define wc = word_count(p); 
showln(wc.count(at(args, 2))); 
close_port(p);

To use this script, you have to start the Slogan interpreter by passing in the file name to parse and word to find the count of occurences:

slogan file.txt word_to_count

Then from the prompt, load "fwc.sn":

slogan> load "fwc";
Compilation

Now that we have developed a substantially large program with two modules and a main script that connects those modules with the outside world, we will look at an important aspect of packaging and distributing software - compilation. So far we have been using the Slogan interpreter. Slogan also has a compiler that can generate machine code for fast execution of programs. We can compile a Slogan script file to platform specific machine code by calling the compile function.

Before running the compiler, you have to point the environment variable SLOGAN_ROOT to the folder where you have downloaded the Slogan source code:

$ export SLOGAN_ROOT=/home/me/slogan

Now we are ready to compile our file word count program:

slogan> compile("fwc", exe = true);

Now you should have an executable named "fwc" along with two dynamically loadable object files (with extension that starts with "o") for the "dict" and "wc" modules. You can package these binary files and distribute them as a standalone application for your platform.
