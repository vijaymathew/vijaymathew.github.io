Interacting with the World

Pure functional programming is somewhat detached from the real world where entities have changing state. To connect a functional program to the real world, some non-functional operations are needed. This chapter covers few such operations - input/output on files and the network, invoking code written in C and compiling Slogan programs into standalone applications for the host operating system.

File and Network IO

A simple way to interface functional programming with the real world is by using files. A file is a sequence of values that is stored external to the program on a permanent storage medium such as a hard disk. A text file is a file containing a sequence of characters. Text files are structured as a sequence of lines, each line ending with the newline character. A binary file contains a sequence of bytes that may represent any type of data. The contents of a binary file has to be interpreted by the program reading it. Both text and binary files mark their end with a special object known as the eof (end-of-file) object.

Reading and writing files

Slogan does all its input and output operations through objects called ports. We read and write files by opening a port on it. A port may be an input port, an output port, or both simultaneously.

A file is read by first opening an input port on it. This is achieved with the open_file_input_port function. On the input file port, we can either call the read_char or read_byte function to fetch a character or a byte respectively. After using the port, it has to be closed by calling close_port. To read a complete text file, we can use the read_all_chars function. To read a complete binary file, call the read_all_bytes function. It will return a list of all bytes in the file. Now let us try to read a text file and display its contents:

// Assuming that you have a file called "hello.txt" with a couple of lines of text in it:
slogan> define in_file = open_file_input_port("hello.txt"); 
slogan> read_all_chars(in_file); 
// => You should see the contents of "hello.txt" here.
slogan> close_port(in_file);

Writing a file is also easy. We open a file output port with open_file_output_port, write characters or bytes using the write_char or write_byte functions and then close the port. We can also call the show or showln functions for writing to a file by passing the port as a keyword argument:

slogan> define out_file = open_file_output_port("out.txt"); 
slogan> showln(port = out_file, "hello from Slogan"); 
slogan> showln(port = out_file, "2 * 3 = ", 2 * 3);            
slogan> close_port(out_file);

The call to close_port is important because it makes sure that the operating system resources associated with the port are released after use. Forgetting to close the port will lead to a resource leak. There is a convenience function called call_with_port that can save us from this trouble. It takes a port and a function as arguments. It calls the function by passing it the port. Once the function returns, call_with_port closes the port. It ensures that the port is closed even if the function returns abnormally by raising an exception. Let us see call_with_port in action by writing a function that makes a copy of a text file:

slogan> function copy(src, dest) 
          call_with_port(open_file_input_port(src), 
           fn (in) call_with_port(open_file_output_port(dest), 
                    fn (out) show(port = out, read_all_chars(in))));

(Slogan has a built-in function - copy_file - that you can use to copy any type of file.)

TCP ports

Files on remote network servers can be read using TCP ports. The following example shows how to read a file hosted on a HTTP server:

slogan> call_with_port(open_tcp_client_port("www.google.com", port_number = 80), 
                       fn(p) { showln(port = p, "GET /search?q=tcp"); 
                               flush_output_port(p); 
                               read_all_chars(p) });

The function flush_output_port makes sure that the request is flushed out from the TCP client ports buffer and pushed to the server. When you run this code, you should see the first results page returned by Google for the query "tcp".

We can use TCP ports to write network servers as well. Here is a simple echo server that sends back whatever the client sends it:

slogan> call_with_port(open_tcp_server_port(9999), 
                       fn(p) let (client = read(p)) 
                              { showln(port = client, read_line(client)); 
                                close_port(client) });

The read function call accepts a new client connection on the server port.

Now you might want to test our echo server by writing a client for it.

Exceptions

Unexpected situations like dividing by zero or opening a nonexistent file can happen during program execution. How to handle these exceptional situations? A functional program can do this by adding cumbersome checks throughout the program. A more elegant way is to use an exception-handling mechanism. Slogan has such a mechanism in the form of the try-catch expression pair. The try expression creates an exception-catching context and the catch expression creates an exception handler. These expressions are complemented by the raise and error functions which are used to raise or throw an exception. Any exception raised in the context of a try is handled by the exception handler in catch. Let us look at a simple example of raising and handling an exception:

slogan> try raise("oops!") 
        catch (e) showln("caught the exception: ", e); 
-> caught the exception: oops!

The try expression has an optional finally clause which is always executed, whether or not an exception is raised. With finally we can guarantee some "cleanup" action happens even if an exception is raised. The following code shows a function that opens a file and reads its entire content. The function handles exception related to file input and also makes sure that the file port is closed before returning:

slogan> function read_file(file_name) 
          let (p = false, s = false) 
           try { p = open_file_input_port(file_name); 
                 s = read_all_chars(p) } 
           catch (e) showln("exception: ", e) 
           finally { if (p) close_port(p); s };

The finally clause makes sure that the port is always closed and the data read into s is returned as the value of the function. If you call read_file with a file that does not exist, it will catch and report the no-such-file-or-directory-exception. (Instead of showln, you can call show_exception(e) which will produce a more descriptive output of standard exceptions.)

FFI

FFI or the Foreign Function Interface is a set of functions that allow Slogan programs to call functions in a shared library. (known as a Dynamic-link Library (DLL) in the Windows world). Shared libraries are usually written in the C programming language. To be callable from Slogan, a C function must follow certain conventions. Its arguments must be of type ___slogan_obj. It has to use type conversion functions defined in the header file "slogan.h" to convert a ___slogan_obj to the appropriate C type. The C types currently supported are: char, bool, short, int, int32, int64, float, double and void pointer. The size of all types except int32 and int64 are system dependent. The C function can directly return all of these types in addition to ___slogan_obj.

The following C program defines two functions that can be called from Slogan. The first function - "add" - expects two integers as arguments and return their sum. The second function - "say_hello" - expects a Slogan string as argument and return a C string:

// demo_lib.c

#include <stdio.h>
#include "slogan.h"

int add(___slogan_obj a, ___slogan_obj b)
{
  int ia, ib;
  ___slogan_obj_to_int(a, &ia);
  ___slogan_obj_to_int(b, &ib);
  return ia + ib;
}

static char *message = "hi from C";
          
char *say_hello(___slogan_obj msg)
{
  char *s;

  ___slogan_obj_to_charstring(msg, &s);
  printf("Slogan says: %s\n", s);
  return message;
}

The following command shows how to call the GNU C Compiler to convert this code into a shared library. Before running the compiler make sure that you have set the SLOGAN_ROOT environment variable to point to the Slogan source folder. 

gcc -Wall -shared -I$SLOGAN_ROOT/src/include -I$SLOGAN_ROOT/platform/gsc/include -fPIC -o demo_lib.so demo_lib.c

On Mac OS X, the command should be:

gcc -dynamiclib -I$SLOGAN_ROOT/src/include -I$SLOGAN_ROOT/platform/gsc/include -Wl,-undefined -Wl,dynamic_lookup -o demo_lib.so demo_lib.c

To load the library into slogan, call the ffi_open function with the library file name as argument. This will give you a handle object that you can use to call functions in the library. A group of functions named in the format - ffi_call_RT_N - is used to invoke C functions. RT denotes the return type of the C function and N specifies the number of arguments the function takes. You can release the library handle by calling ffi_close. The following program shows how to load and use our demo_lib shared library:

slogan> define clib = ffi_open("./demo_lib.so"); 
slogan> ffi_call_int_2(clib, "add", 10, 20); 
=> 30
slogan> ffi_call_char_string_1(clib, "say_hello", "hi from Slogan"); 
-> Slogan says: hi from Slogan
=> hi from C
slogan> ffi_close(clib);

Modules and Compilation

As a program grows in size, we will need some way to organize it. Writing the program as one big monolithic whole can be confusing. A better way is to partition the program into logical units, each of which implements a set of operations that are related in some way. Each logical unit has two parts, an interface and an implementation. Only the interface is visible from outside the logical unit. A logical unit may use others as part of its implementation.

We can use closures to group together related operations into an entity that has an interface and an implementation. The following example shows a dictionary module that can be used to associate a word with some value. The dictionary module exports two operations to the outside world:

  1. put - adds a word and its associated value to the dictionary.
  2. get - retrieves the value associated with a word.

Exactly how these operations are implemented is hidden from the world. Someone who uses the dictionary module only need to know about the functionality provided by the two exported operations. The following implementation of dictionary uses the built-in hashtable data structure to associate words with values. This time let us write and save the code in a source file called "dict.sn":

// file: dict.sn
function dict() 
  { define table = make_hashtable(string_hash, string_is_eq); 
    function put(w, v) hashtable_set(table, w, v); 
    function get(w) hashtable_at(table, w); 

    fn (message) 
      match (message) 
       !put -> put, 
       !get -> get };

To use the dictionary module, we have to first load the file "dict.sn" into the REPL:

slogan> load "dict"; 

slogan> define d = dict(); 
slogan> d.put("name", "sam"); 
slogan> d.put("age", 2);          
slogan> showln(d.get("name"), " is ", d.get("age"), " years old."); 
-> sam is 2 years old.

We will write a new module that uses the features exported by dictionary. This is a module that can count the words read from a text input port:

// file: word_count.sn 
load "dict"; 

function word_count(port) 
  { define wd = let init_words_dict (words = string_split(read_all_chars(port)), 
                                     wd = dict()) 
                 if (is_empty(words)) wd 
                 else let (count = wd. get(head(words))) 
                       { if (not(count)) count = 0; 
                         wd.put(head(words), count + 1); 
                         init_words_dict(tail(words), wd) }; 

    function count(word) wd.get(word); 

    fn (message) match (message) !count -> count };

The word_count module presents a new feature called named let which allows a let block to be named and used like a recursive function.

Let us complete this section by writing a script that uses the word_count module to find the number of occurrences of words in a file. The file to read is passed as a command line argument to the script:

// file: file_word_count.sn 

load "word_count"; 

define args = command_line(); 
define p = open_file_input_port(at(1, args)); 
define wc = word_count(p); 

function find_count() 
  { show("Enter word to count: "); 
    let (w = read_line()) 
     if (not(w == ":q")) 
      { showln(wc.count(w)); 
        find_count() } }; 

find_count(); 
close_port(p);

To use this script, you have to start the Slogan interpreter by passing in the file name to parse:

slogan file.txt

Then from the prompt, load "file_word_count.sn":

slogan> load "file_word_count";

The script will wait for you to enter a word and it will report back its number of occurrences in the file. It will continue this loop till you enter ":q", which is a command that the script interprets as "quit".

Compilation

Now that we have developed a substantially large program with two modules and a main script that connects those modules with the outside world, we will look at an important aspect of packaging and distributing software - compilation.

So far we have been using the Slogan interpreter. Slogan also has a compiler that can generate machine code for fast execution of programs. We can compile a Slogan script file to platform specific machine code by calling the compile function.

Before running the compiler, you have to point the environment variable SLOGAN_ROOT to the folder where you have downloaded the Slogan source code:

$ export SLOGAN_ROOT=/home/me/slogan

We are all set to compile our "file word count" program:

slogan> compile("file_word_count", exe = true);

Now you should have an executable named "file_word_count" along with two dynamically loadable object files (with an extension that starts with an "o") for the "dict" and "word_count" modules. You can package these binary files and distribute them as a standalone application for your platform.
